Q41객관식 · DCL GRANT/REVOKE

사용자에게 테이블 조회 권한을 부여하거나 회수하는 명령(DCL)으로 짝지어진 것은?

  1. COMMIT / ROLLBACK
  2. GRANT / REVOKE
  3. CREATE / DROP
  4. INSERT / DELETE
정답 확인
② GRANT / REVOKE
GRANT는 권한 부여, REVOKE는 회수로 둘 다 DCL(데이터 제어어)이다. SI에서 고객사 DB 계정에 최소권한 원칙으로 SELECT만 GRANT하는 식으로 보안을 관리하며, 퇴사·역할 변경 시 REVOKE로 접근을 정리한다.
Q42객관식 · TCL SAVEPOINT

SAVEPOINT에 대한 설명으로 옳은 것은?

  1. 트랜잭션을 즉시 영구 저장(커밋)한다
  2. 테이블 구조를 백업한다
  3. 사용자 권한을 저장한다
  4. 트랜잭션 내 특정 지점을 표시해, 그 지점까지만 부분 롤백할 수 있게 한다
정답 확인
④ 트랜잭션 내 특정 지점을 표시해, 그 지점까지만 부분 롤백할 수 있게 한다
SAVEPOINT는 긴 트랜잭션 중간에 지점을 찍고 ROLLBACK TO 지점으로 일부만 되돌린다. SI 대량 배치에서 앞 단계 처리는 살리고 오류 난 단계만 취소하는 세밀한 제어에 쓰여 재처리 비용을 줄인다.
Q43객관식 · INTERSECT/MINUS

두 SELECT 결과에서 "양쪽 모두에 존재하는 공통 행"만 반환하는 집합연산은? (오라클 기준)

  1. UNION
  2. UNION ALL
  3. INTERSECT
  4. MINUS
정답 확인
③ INTERSECT
INTERSECT는 교집합, MINUS는 차집합(앞 결과에서 뒤 결과를 뺌)이다. SI에서 "두 기간 모두 거래한 고객"(INTERSECT), "작년엔 있었지만 올핸 없는 고객"(MINUS)처럼 명단 대사(對査) 요건에 직결된다.
Q44객관식 · NULL 비교

comm이 NULL인 행을 찾으려 한다. 올바른 조건은?

  1. WHERE comm IS NULL
  2. WHERE comm = NULL
  3. WHERE comm == NULL
  4. WHERE comm != 0
정답 확인
① WHERE comm IS NULL
NULL은 '알 수 없음'이라 = NULL 같은 비교는 참도 거짓도 아닌 UNKNOWN이 되어 아무 행도 안 걸린다. 반드시 IS NULL/IS NOT NULL을 써야 한다. SI에서 미입력 데이터 추출 시 가장 흔히 틀리는 함정이다.
Q45객관식 · 인덱스 사용조건

emp_id 컬럼에 인덱스가 있어도 인덱스를 타지 못하게(무력화) 만들 가능성이 높은 조건은?

  1. WHERE emp_id = 100
  2. WHERE emp_id BETWEEN 100 AND 200
  3. WHERE SUBSTR(emp_id, 1, 1) = '1'
  4. WHERE emp_id IN (100, 200, 300)
정답 확인
③ WHERE SUBSTR(emp_id, 1, 1) = '1'
인덱스 컬럼을 함수로 가공하면 원본 값 기준의 인덱스를 쓸 수 없어 대개 풀스캔이 된다(함수기반 인덱스가 없다면). SI 성능 장애의 단골 원인으로, 컬럼은 가공하지 말고 상수 쪽을 변형하는 방향으로 조건을 짜야 한다.
Q46객관식 · UNIQUE 제약

UNIQUE 제약과 PRIMARY KEY의 차이로 옳은 것은?

  1. UNIQUE는 NULL을 허용할 수 있고, 한 테이블에 여러 개 지정할 수 있다
  2. UNIQUE는 NULL을 절대 허용하지 않는다
  3. 한 테이블에 UNIQUE는 하나만 지정할 수 있다
  4. UNIQUE는 중복 값을 허용한다
정답 확인
① UNIQUE는 NULL을 허용할 수 있고, 한 테이블에 여러 개 지정할 수 있다
UNIQUE는 유일성만 보장하고 (DBMS마다 차이는 있으나) NULL을 허용하며 테이블에 여러 개 걸 수 있다. PK는 테이블당 하나이고 NOT NULL이다. SI에서 이메일·사업자번호처럼 식별자는 아니지만 중복은 막아야 하는 컬럼에 UNIQUE를 쓴다.
Q47쿼리작성 · 인라인뷰

아래 스키마에서 요구사항을 만족하는 SQL을 작성하라. (인라인뷰 사용)

스키마
EMP(emp_id, name, dept_id, salary)
DEPT(dept_id, dept_name)

요구사항
부서별 평균 급여를 구한 뒤, 그 평균이 4000 이상인 부서의 dept_name과 평균급여(avg_sal)를 조회하라.
모범답안 확인
모범답안
SELECT d.dept_name, t.avg_sal
FROM   (SELECT dept_id, AVG(salary) AS avg_sal
        FROM   EMP
        GROUP  BY dept_id
        HAVING AVG(salary) >= 4000) t
JOIN   DEPT d ON t.dept_id = d.dept_id;
FROM 절 서브쿼리(인라인뷰)로 집계 결과를 임시 테이블처럼 만든 뒤 부서명과 조인하는 패턴이다. "집계한 결과에 다시 코드명을 붙이는" 2단계 처리는 SI 통계 리포트에서 매우 자주 나오는 구조다.
Q48쿼리작성 · UNION

아래 스키마에서 요구사항을 만족하는 SQL을 작성하라.

스키마
ONLINE_ORDER(order_id, cust_id, amount)
OFFLINE_ORDER(order_id, cust_id, amount)

요구사항
온라인·오프라인 주문을 하나로 합쳐 order_id, cust_id, amount를 조회하되, 완전히 동일한 행은 한 번만 나오게 하라.
모범답안 확인
모범답안
SELECT order_id, cust_id, amount FROM ONLINE_ORDER
UNION
SELECT order_id, cust_id, amount FROM OFFLINE_ORDER;
두 SELECT의 컬럼 개수·타입·순서를 맞추고 중복 제거가 필요하므로 UNION을 쓴다(중복 허용이면 UNION ALL). 채널별로 분리 저장된 데이터를 통합 조회하는 것은 SI 통합 리포트의 전형적 요건이다.
Q49쿼리작성 · 다중 JOIN

아래 스키마에서 요구사항을 만족하는 SQL을 작성하라.

스키마
EMP(emp_id, name, dept_id)
DEPT(dept_id, dept_name, loc_id)
LOCATION(loc_id, city)

요구사항
각 사원의 name, 부서명(dept_name), 근무 도시(city)를 함께 조회하라.
모범답안 확인
모범답안
SELECT e.name, d.dept_name, l.city
FROM   EMP e
JOIN   DEPT d     ON e.dept_id = d.dept_id
JOIN   LOCATION l ON d.loc_id  = l.loc_id;
3개 테이블을 공통 키로 연쇄 조인하며 각 ON 조건을 정확히 잇는 것이 채점 포인트다. 정규화된 SI 스키마에서는 화면 하나를 채우려 여러 코드성 테이블을 이어 조인하는 일이 일상이라, 조인 경로 파악이 실무 핵심 역량이다.
Q50단답 · HAVING/COUNT

ORDERS 테이블의 cust_id별 주문 건수가 A=1건, B=3건, C=2건, D=1건이다. SELECT cust_id FROM ORDERS GROUP BY cust_id HAVING COUNT(*) >= 2 의 결과 행수는? 숫자로 답하라.

모범답안 확인
모범답안 · 2
주문 2건 이상인 고객은 B(3건)·C(2건) 두 명이므로 2행이다. HAVING은 그룹 집계 후 조건을 적용한다는 점을 결과 건수로 확인하는 문제로, SI에서 "재구매 고객 수" 같은 지표 검증에 직결되는 감각이다.

이 세트 우선복습 3줄

  1. NULL 비교 함정 — = NULL은 UNKNOWN이 되어 한 행도 안 걸린다. NULL 탐색은 반드시 IS NULL/IS NOT NULL.
  2. 인덱스 무력화 — 인덱스 컬럼을 SUBSTR(col,…)처럼 함수로 가공하면 풀스캔. 컬럼은 두고 상수 쪽을 변형하라.
  3. UNIQUE vs PK — UNIQUE는 NULL 허용 + 테이블당 여러 개 가능, PK는 NOT NULL + 테이블당 하나. UNION은 중복 제거, UNION ALL은 유지.
