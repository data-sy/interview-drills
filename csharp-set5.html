Q41객관식 · ASP.NET vs Spring

서버 웹 프레임워크의 자바↔C# 대비로 옳은 것은?

  1. .NET에는 웹 프레임워크가 없어 서블릿을 직접 구현해야 한다
  2. 자바 진영의 Spring MVC / Spring Boot에 대응해 C#/.NET은 ASP.NET Core MVC / Web API로 웹·REST 서비스를 만든다
  3. ASP.NET은 자바로 작성하는 프레임워크다
  4. Spring과 ASP.NET은 모두 클라이언트 전용 UI 라이브러리다
정답 확인
② 자바 Spring MVC/Boot ↔ ASP.NET Core MVC / Web API
자바의 Spring MVC/Boot에 대응하는 것이 .NET의 ASP.NET Core MVC(뷰 렌더)와 Web API(REST)다. 둘 다 컨트롤러·라우팅·의존성 주입(DI)을 프레임워크 차원에서 지원한다. SI에서 백엔드 스택이 자바냐 .NET이냐에 따라 갈리지만 MVC·DI 개념은 그대로 옮겨 이해할 수 있다.
Q42객관식 · IEnumerable vs Iterable

반복 가능 컬렉션 추상화의 자바↔C# 대비로 옳은 것은?

  1. 자바 Iterable/Iterator에 대응해 C#은 IEnumerable/IEnumerator이며, foreach는 이 인터페이스를 통해 순회한다
  2. C#의 foreach는 인터페이스와 무관하게 배열에만 동작한다
  3. IEnumerable은 자바 인터페이스이고 C#은 Iterable을 구현한다
  4. C#에는 이터레이터 개념이 없어 인덱스 for문만 쓴다
정답 확인
① Iterable/Iterator ↔ IEnumerable/IEnumerator
자바 Iterable/Iterator(hasNext/next)에 대응하는 C#은 IEnumerable/IEnumerator(MoveNext/Current)이며, foreach가 이를 통해 돈다. C#은 yield return으로 이터레이터를 쉽게 구현한다. 대용량 결과를 한 번에 메모리에 올리지 않고 스트리밍 순회할 때 이 추상화가 기반이 된다.
Q43객관식 · nameof/typeof

이름·타입 정보를 얻는 방식의 자바↔C# 대비로 옳은 것은?

  1. C#에는 타입 정보를 얻는 수단이 전혀 없다
  2. nameof는 런타임 문자열 파싱이라 오타를 못 잡는다
  3. 자바 getClass()와 C# nameof는 완전히 같은 런타임 동작이다
  4. 자바는 리플렉션(getClass().getName())으로 이름을 얻지만, C#은 컴파일타임에 안전한 nameof(x)와 typeof(T)를 제공한다
정답 확인
④ 자바 리플렉션 ↔ C# 컴파일타임 nameof(x)/typeof(T)
자바는 주로 리플렉션(obj.getClass().getName())으로 이름을 얻지만, C#은 nameof(변수/멤버)로 컴파일타임에 이름 문자열을 안전하게 얻고 typeof(T)로 Type을 얻는다. nameof는 리팩터링 시 이름이 바뀌면 함께 갱신돼 오타를 막는다. 로그·검증 메시지·프로퍼티명 하드코딩을 없애 유지보수성을 높인다.
Q44객관식 · 정적 클래스

인스턴스를 만들지 않는 유틸리티 묶음의 자바↔C# 대비로 옳은 것은?

  1. C#의 static class는 인스턴스를 만들 수 있고 인스턴스 필드도 가진다
  2. 자바에는 정적 메서드만 모으는 방식이 없다
  3. 자바는 생성자를 private로 막은 유틸 클래스로 정적 메서드를 모으고, C#은 static class로 인스턴스화 자체를 언어가 금지한다
  4. static class는 자바 키워드이고 C#엔 없다
정답 확인
③ 자바 private 생성자 유틸 ↔ C# static class로 인스턴스화 금지
자바는 관례적으로 생성자를 private로 막은 유틸리티 클래스에 static 메서드를 모으지만, C#은 static class로 아예 인스턴스화·인스턴스 멤버를 컴파일러가 금지한다(정적 멤버만 허용). 공용 헬퍼(포맷터·계산기)를 안전하게 묶을 때 실수로 인스턴스를 만드는 걸 원천 차단한다.
Q45객관식 · 값/참조 전달

참조 타입 인자 전달 동작의 자바↔C# 대비로 옳은 것은?

  1. 자바는 객체를 통째로 복사해 넘기고 C#은 원본을 그대로 넘긴다
  2. 두 언어 모두 기본은 참조의 "값"(주소 복사)을 넘기므로 메서드 안에서 필드 변경은 반영되지만 인자를 새 객체로 재대입해도 호출부엔 반영되지 않는다
  3. 두 언어 모두 참조 타입 인자를 재대입하면 호출부 변수도 바뀐다
  4. C#은 참조 전달만 되고 자바만 값 전달이라 동작이 정반대다
정답 확인
② 둘 다 참조값(주소) 복사 전달 — 필드 변경은 반영, 재대입은 미반영
자바·C# 모두 기본적으로 참조 타입은 "참조값(주소)"을 복사해 전달한다. 그래서 넘어온 객체의 필드를 바꾸면 호출부에 보이지만, 매개변수에 new로 다른 객체를 대입해도 호출부 변수는 그대로다. C#은 여기에 ref로 진짜 참조 전달까지 추가로 지원한다. 이 미묘함을 모르면 "왜 바뀐 게 안 보이지" 류 버그를 낳는다.
Q46객관식 · LINQ 지연 실행

LINQ 지연 실행(deferred execution)의 자바 Stream 대비 설명으로 옳은 것은?

  1. 자바 Stream이 최종 연산 전엔 실행되지 않듯, LINQ 쿼리도 ToList/foreach 등으로 열거하기 전까진 실행되지 않아 원본이 바뀌면 결과도 달라진다
  2. LINQ 쿼리는 정의하는 즉시 무조건 실행돼 결과가 그 자리에서 고정된다
  3. 자바 Stream은 지연이지만 LINQ는 절대 지연되지 않는다
  4. 지연 실행은 성능과 무관한 순수 문법 규칙일 뿐 동작에 영향이 없다
정답 확인
① LINQ도 열거 시점에 실행(지연) — 원본이 바뀌면 결과도 달라진다
LINQ는 자바 Stream의 최종 연산처럼, Where/Select만 걸어 둔 쿼리를 ToList()·foreach로 열거할 때 비로소 실행한다(지연 실행). 그래서 정의 후 원본 컬렉션이 바뀌면 나중 열거 결과가 달라지는 함정이 있다. 결과를 고정하려면 즉시 ToList()로 구체화하는 게 실무 방어 코드다.
Q47객관식 · null 조건부 ?.

null 안전 접근의 자바↔C# 대비로 옳은 것은?

  1. C#의 ?. 는 null이어도 강제로 메서드를 호출해 예외를 던진다
  2. ?. 는 자바 연산자이고 C#은 지원하지 않는다
  3. 자바는 Optional/명시적 null 체크로 NPE를 피하고, C#은 ?. (null 조건부 연산자)로 null이면 호출을 건너뛰고 null을 반환한다
  4. 자바에는 null 안전 수단이 전혀 없다
정답 확인
③ 자바 Optional/null 체크 ↔ C# ?. (null이면 건너뛰고 null 반환)
자바는 Optional이나 if(x!=null) 체크로 NullPointerException을 피하고, C#은 obj?.Name처럼 ?. 로 앞이 null이면 뒤를 평가하지 않고 null을 반환한다. ?. + ??를 엮어 obj?.Name ?? "기본"으로 안전하게 쓴다. 외부 연동 데이터의 중첩 null을 방어할 때 코드가 짧고 견고해진다.
Q48객관식 · 형변환 is/as

타입 검사·변환의 자바↔C# 대비로 옳은 것은?

  1. C#의 as는 변환 실패 시 예외를 던져 자바 캐스트와 동일하다
  2. 자바 instanceof와 C# is는 둘 다 변환까지 수행해 캐스트가 필요 없다는 점만 같다
  3. is/as는 자바 연산자이고 C#은 instanceof를 쓴다
  4. 자바 instanceof + 캐스트에 대응해 C#은 is(타입 검사, 패턴 매칭)와 as(실패 시 null 반환 캐스트)를 제공한다
정답 확인
④ 자바 instanceof+캐스트 ↔ C# is(패턴 매칭)/as(실패 시 null)
자바는 instanceof로 검사 후 캐스트하지만, C#은 is로 검사(패턴 매칭 if (o is Dog d))하고 as로 실패 시 예외 대신 null을 돌려주는 안전 캐스트를 제공한다. 직접 캐스트 (Dog)o는 실패 시 InvalidCastException을 던진다. 다형적 데이터를 분기 처리할 때 as/is로 방어적으로 다룬다.
Q49코드읽기 · struct 복사 동작

다음 C# 코드의 출력은? 자바에는 값 형식 struct가 없다는 점과 대비하며 읽어라.

struct Point { public int X; }
Point a = new Point { X = 1 };
Point b = a;      // 값 복사
b.X = 99;
Console.WriteLine($"{a.X},{b.X}");
모범답안 확인
모범답안 · 1,99
struct는 값 형식이라 b = a에서 내용이 통째로 복사된다. 그래서 b.X를 바꿔도 a.X는 1로 남는다. 만약 Point가 class(참조 형식)였다면 a,b가 같은 객체를 가리켜 둘 다 99가 됐을 것이다. 자바는 사용자 정의 값 형식이 없어 항상 후자처럼 동작하므로, C# struct의 복사 시맨틱을 모르면 이관 버그가 난다.
Q50단답 · record

자바 14+의 record가 불변 데이터 캐리어와 equals/hashCode/toString을 자동 생성하듯, C#에서 이에 대응하는 참조형 불변 데이터 타입 키워드를 쓰고 자동 생성되는 것 하나를 쓰시오.

모범답안 확인
모범답안 · record
record (C# 9+) — 값 기반 Equals/GetHashCode/ToString과 with 식(비파괴적 복사)을 자동 생성한다.

자바 record와 C# record 모두 불변 데이터를 간결히 선언하고 값 기반 동등성(Equals/GetHashCode)·ToString을 자동 생성한다. C# record는 추가로 with 식으로 일부 값만 바꾼 복사본을 만든다. DTO·조회 결과 모델처럼 데이터를 담기만 하는 타입을 보일러플레이트 없이 정의해 유지보수를 줄인다.

이 세트 우선복습 3줄

  1. 참조 타입 인자 전달 — 자바·C# 모두 기본은 참조값(주소) 복사. 객체 필드 변경은 호출부에 보이지만, 매개변수에 new로 재대입한 건 안 보인다(C#은 ref로 진짜 참조 전달 추가).
  2. LINQ 지연 실행 — Where/Select만 걸면 실행 안 되고 ToList()·foreach로 열거할 때 실행. 정의 후 원본이 바뀌면 결과가 달라지니, 고정하려면 즉시 ToList().
  3. is / as 형변환 — is는 검사(패턴 매칭 o is Dog d), as는 실패 시 null 반환. 직접 캐스트 (Dog)o는 실패 시 InvalidCastException.
