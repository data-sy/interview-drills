Q21구두 · 설명형 · 추상클래스 vs 인터페이스

추상클래스와 인터페이스의 차이와, 언제 무엇을 쓰는지 설명해보세요.

모범답변 요지
  • 추상클래스: 공통 필드·구현을 물려주며 "is-a" 관계, 단일 상속
  • 인터페이스: 구현 규약(기능)을 정의, 다중 구현 가능, "할 수 있다(can-do)" 관계
  • 공통 상태·기본 구현을 공유하려면 추상클래스, 서로 다른 타입에 공통 기능만 강제하려면 인터페이스
  • 실무 연결: 외부 연동 규격을 인터페이스로 정의하면 구현체를 갈아끼워도 호출부가 안정적이라, SI 연동 개발에서 유연성을 준다는 점 언급
Q22구두 · 설명형 · equals vs ==

자바에서 == 와 equals()의 차이를 설명해보세요.

모범답변 요지
  • ==: 기본형은 값 비교, 참조형은 주소(동일 객체인지) 비교
  • equals(): 내용(값) 비교, 기본은 == 와 같지만 String 등은 오버라이드되어 내용 비교
  • equals를 재정의하면 hashCode도 함께 재정의해야 함
  • 실무 연결: 문자열 비교를 ==로 하면 값이 같아도 false가 나오는 버그가 흔해, 참조/값 비교 구분이 실무 오류 예방의 기본임을 강조
Q23구두 · SI 실무형 · 납기/일정

납기가 촉박한데 맡은 기능을 제때 끝내기 어려워 보입니다. 어떻게 하겠습니까?

모범답변 요지
  • 지연을 숨기지 않고 조기에 리더·PM에 상황과 예상 지연을 공유
  • 기능을 필수/부가로 나눠 우선순위대로 처리(MVP 우선)
  • 도움 요청·업무 재분배 등 팀 차원 대안 제시
  • 무리한 야근 대신 품질을 지키는 현실적 조정
  • 실무 연결: SI에서 납기 리스크를 늦게 알리는 것이 가장 큰 사고라, 투명한 조기 공유와 우선순위 협의가 신뢰의 핵심임을 강조
Q24구두 · 설명형 · JOIN 종류

INNER JOIN과 LEFT OUTER JOIN의 차이를 설명해보세요.

모범답변 요지
  • INNER JOIN: 양쪽 테이블에 조인 조건이 일치하는 행만 반환
  • LEFT OUTER JOIN: 왼쪽 테이블은 전부 유지, 오른쪽에 없으면 NULL로 채움
  • 매칭 안 되는 데이터도 봐야 할 때 OUTER, 교집합만 필요하면 INNER
  • 실무 연결: "주문은 있는데 배송정보가 없는 건"처럼 누락 데이터 점검이 필요한 통계·검증 쿼리에서 OUTER JOIN이 필수임을 예로 언급
Q25구두 · 설명형 · 세션 vs 쿠키

세션과 쿠키의 차이를 설명하고, 로그인 상태 유지에 어떻게 쓰이는지 말해보세요.

모범답변 요지
  • 쿠키: 클라이언트(브라우저)에 저장, 위변조·노출 위험, 용량 제한
  • 세션: 서버에 저장하고 클라이언트는 세션ID만 쿠키로 보관, 상대적으로 안전
  • 로그인 정보 같은 민감 데이터는 세션에, 세션ID를 쿠키로 주고받음
  • 실무 연결: 사용자 인증을 다루는 업무 시스템에서 민감정보를 쿠키에 그대로 담으면 보안 사고가 되므로 저장 위치 구분이 중요함을 강조
Q26구두 · SI 실무형 · 배포/장애 대응

운영 중인 시스템에서 장애가 발생했다는 연락을 받았습니다. 어떤 순서로 대응하겠습니까?

모범답변 요지
  • 먼저 영향 범위·증상을 파악하고 관련자에게 상황 공유
  • 원인 분석보다 서비스 복구(롤백·긴급 조치)를 우선할지 판단
  • 로그·모니터링으로 원인 추적, 조치 후 재발 방지책 마련
  • 대응 과정을 기록해 사후 공유(포스트모템)
  • 실무 연결: 고객사 운영 시스템은 다운타임이 곧 손실이라, "복구 우선 → 원인 분석 → 재발 방지" 순서와 침착한 보고 체계를 강조
Q27구두 · 설명형 · 스택 vs 큐

스택과 큐의 차이를 설명하고 각각 어디에 쓰이는지 예를 들어보세요.

모범답변 요지
  • 스택: LIFO(후입선출), 예) 함수 호출·되돌리기(undo)·괄호 검사
  • 큐: FIFO(선입선출), 예) 작업 대기열·메시지 큐·프린터 스풀
  • 자료를 넣고 빼는 순서 규칙이 반대
  • 실무 연결: 배치·비동기 작업 처리에서 요청을 순서대로 처리하는 큐 개념이 자주 쓰여, 자료구조 선택이 처리 순서를 결정함을 언급
Q28구두 · 설명형 · DI/IoC

Spring의 의존성 주입(DI)과 제어의 역전(IoC)이 무엇인지 설명해보세요.

모범답변 요지
  • IoC: 객체 생성·생명주기 제어를 개발자가 아닌 컨테이너가 담당
  • DI: 필요한 객체를 직접 new 하지 않고 외부에서 주입받음(생성자·필드 등)
  • 결합도가 낮아지고 테스트(모의 객체 주입)·교체가 쉬워짐
  • 실무 연결: 구현체 교체가 잦은 SI 연동 개발에서 DI로 결합을 낮추면 유지보수·단위 테스트가 쉬워진다는 점 언급
Q29구두 · SI 실무형 · 테스트/품질

본인이 만든 기능이 제대로 동작하는지 어떻게 확인하고 품질을 보장하나요?

모범답변 요지
  • 정상 케이스뿐 아니라 경계값·예외·빈 값 등 다양한 케이스로 검증
  • 단위 테스트 작성, 실제 데이터·화면 흐름으로 통합 확인
  • 남의 코드/내 코드 모두 리뷰받고, 재현 절차를 기록
  • 실무 연결: 고객 인수 후 발견되는 결함은 비용이 크므로, 개발 단계에서 다양한 케이스를 스스로 검증하는 습관이 SI 품질의 기본임을 강조
Q30구두 · 인성/동기형 · 약점

본인의 약점은 무엇이고 어떻게 보완하고 있나요?

모범답변 요지
  • 치명적이지 않은 실제 약점을 솔직히 제시(예: 완벽주의로 속도가 느림)
  • 그 약점을 인지하고 개선하려는 구체적 노력을 함께 설명
  • 개선 중이라는 성장 지향적 태도로 마무리
  • 실무 연결: 약점을 감추기보다 자기 인식과 개선 의지를 보여주는 것이 핵심이며, SI 협업에서 피드백 수용력과 연결되는 답을 강조

이 세트 우선복습 3줄

  1. 추상클래스 vs 인터페이스 — 추상클래스는 공통 구현을 물려주는 "is-a"(단일 상속), 인터페이스는 규약만 강제하는 "can-do"(다중 구현). 상태 공유는 추상클래스, 이질적 타입에 기능 강제는 인터페이스.
  2. == vs equals() — ==는 참조형에서 주소 비교, equals()는 내용 비교(String은 오버라이드됨). 문자열을 ==로 비교하면 값이 같아도 false가 나는 단골 버그.
  3. 세션 vs 쿠키 — 쿠키는 브라우저 저장(노출·위변조 위험), 세션은 서버 저장이고 클라이언트는 세션ID만 쿠키로 보관. 민감정보는 세션에 두는 것이 원칙.
