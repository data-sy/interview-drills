기본 · 자료구조

01배열 Array

고정 길이. 크기 변경 불가 → 가변이 필요하면 List를 쓴다. Arrays 유틸이 실무에서 자주 나온다.

면접 포인트
  • Arrays.asList()는 고정 크기 뷰라 add/remove 시 UnsupportedOperationException. 가변이 필요하면 new ArrayList<>(...)로 감싼다.
  • 배열 길이는 .length(필드), String은 .length()(메서드), List는 .size() — 헷갈리기 쉽다.

02컬렉션 List · Map · Set

실무 8할은 이 셋. 인터페이스 타입으로 선언하고 구현체로 생성하는 습관이 중요하다.

List — 순서 O, 중복 O

Map — key-value, key 중복 X

Set — 중복 X

면접 포인트
  • ArrayList vs LinkedList: ArrayList는 인덱스 조회 O(1)·중간 삽입 느림, LinkedList는 조회 O(n)·중간 삽입 빠름. 실무는 대부분 ArrayList.
  • HashMap vs TreeMap vs LinkedHashMap: 순서 없음 / 키 정렬 / 입력 순서. 정렬이나 순서가 필요 없으면 HashMap(가장 빠름).
  • HashMap의 key로 쓰는 객체는 equals()·hashCode()를 반드시 함께 재정의해야 한다.

03Stack · Queue · Deque

LIFO/FIFO 자료구조. 실무·코테 모두 Deque(ArrayDeque)를 스택·큐 겸용으로 쓰는 걸 권장한다.

면접 포인트
  • 구형 Stack은 Vector 기반이라 동기화 오버헤드가 있다 → ArrayDeque가 더 빠르고 권장된다.
  • Queue에서 add/remove는 실패 시 예외, offer/poll은 실패 시 false/null. 실무는 offer/poll이 안전.

04정렬 · 순회 Comparator

정렬 기준을 Comparator로 지정하는 패턴은 실무 필수. 람다·메서드 참조와 함께 자주 쓴다.

면접 포인트
  • for-each 순회 중 컬렉션을 remove하면 ConcurrentModificationException. → Iterator.remove()나 removeIf()를 쓴다.
  • Comparable vs Comparator: Comparable은 클래스 자신의 기본 정렬 기준(compareTo), Comparator는 외부에서 주입하는 별도 기준.
언어 기능

05String · StringBuilder

String은 불변(immutable). 반복 문자열 조합은 반드시 StringBuilder를 쓴다.

면접 포인트
  • 문자열 비교는 ==(주소)가 아니라 equals()(값). NPE 방지로 "상수".equals(변수) 순서를 쓰기도 한다.
  • String vs StringBuilder vs StringBuffer: String 불변 / StringBuilder 가변·비동기(빠름) / StringBuffer 가변·동기화(스레드 안전). 단일 스레드면 StringBuilder.

06클래스 · VO / DTO

계층 간 데이터 운반 객체. SI에선 getter/setter 풀버전을, 최근 프로젝트는 record를 쓴다.

전통적 VO (getter/setter)

record (Java 16+) — 위 코드를 한 줄로모던

면접 포인트
  • DTO를 컬렉션 key나 비교에 쓰면 equals()·hashCode()를 반드시 세트로 재정의(하나만 하면 버그). record는 자동 처리.
  • VO vs DTO vs Entity: VO는 값 자체(불변 지향), DTO는 계층 간 전송용, Entity는 DB 테이블 매핑. 용어를 섞어 쓰는 현장도 많으니 문맥으로 판단.

07인터페이스 · 상속

다형성의 뼈대. 서비스 계층을 인터페이스로 추상화하는 패턴이 SI/SM에 흔하다.

면접 포인트
  • 인터페이스 vs 추상클래스: 인터페이스는 "할 수 있다(구현 다중)", 추상클래스는 "이다(단일 상속·공통 상태)". 다중 구현 가능 여부가 핵심 차이.
  • Java 8부터 인터페이스에 default·static 메서드 허용 → 추상클래스와 경계가 흐려짐. 그래도 상태(필드)는 추상클래스만.
  • 오버라이딩(재정의, 시그니처 동일) vs 오버로딩(중복정의, 파라미터 다름) 구분.

08Enum

고정된 상수 집합. 코드값·상태값 관리에 쓰고, 필드·메서드를 가질 수 있다.

면접 포인트
  • 상태·코드 상수를 int/String 상수로 흩어 두는 대신 enum으로 묶으면 타입 안전 + 값·라벨을 함께 관리.
  • ordinal()은 선언 순서에 의존하니 DB 저장값으로 쓰면 위험 → 별도 코드 필드(step 등)를 두는 게 안전.

09예외처리 Exception

try-catch-finally, try-with-resources, 커스텀 예외. 자원 해제와 예외 계층 이해가 핵심.

구분Checked 예외Unchecked 예외
상속ExceptionRuntimeException
처리 강제컴파일러가 강제(try/throws 필수)강제 안 함
예시IOException, SQLExceptionNPE, IllegalArgument, Biz예외
면접 포인트
  • Checked vs Unchecked: RuntimeException 계열(unchecked)은 try/throws 강제 안 됨. 실무 비즈니스 예외는 보통 RuntimeException을 상속.
  • finally는 return이 있어도 실행된다. 단 자원 해제는 try-with-resources가 더 안전(누수·순서 실수 방지).
  • 예외를 삼키지(빈 catch) 말고 로깅하거나 원인(cause)을 담아 다시 던진다.

10제네릭 Generics

타입을 파라미터로. 컴파일 시점 타입 안전성 + 캐스팅 제거.

면접 포인트
  • 제네릭은 컴파일 타임에만 타입 체크하고 런타임엔 소거(type erasure)된다 → new T() 불가.
  • PECS 원칙: Producer Extends(읽기 ? extends), Consumer Super(쓰기 ? super).
모던 자바 (8+)

11람다 · 스트림 Stream

컬렉션 처리를 선언형으로. 실무에서 for문을 대체하는 빈도가 매우 높다.

면접 포인트
  • 스트림은 지연 연산: 중간 연산(filter/map)은 최종 연산(collect/count 등)이 호출될 때 실행된다.
  • 람다는 함수형 인터페이스(추상 메서드 1개)를 구현: Function, Predicate, Consumer, Supplier.
  • 메서드 참조 Emp::name은 e -> e.name()의 축약.

12Optional

null을 명시적으로 다루는 래퍼. NPE 방지 + "값이 없을 수 있음"을 타입으로 표현.

면접 포인트
  • get()은 비었을 때 예외 → orElse/orElseThrow/ifPresent로 안전하게.
  • orElse는 항상 인자를 평가, orElseGet은 비었을 때만 평가(비용 큰 기본값이면 orElseGet).
  • Optional은 반환 타입용. 필드·파라미터로 남발하지 않는 게 관례.

13날짜 · 시간 (java.time)

Java 8의 java.time 패키지. 구형 Date/Calendar는 지양.

면접 포인트
  • LocalDate(날짜) / LocalTime(시간) / LocalDateTime(둘 다) / ZonedDateTime(시간대) 구분.
  • java.time 객체는 모두 불변 → plusDays 등은 원본을 바꾸지 않고 새 객체를 반환한다.
백엔드 실무

14Map 관용구

집계·카운팅·그룹핑에서 반복되는 Map 패턴. 알아두면 코드가 확 짧아진다.

면접 포인트
  • getOrDefault·merge·computeIfAbsent는 "없으면 초기화" 분기(if null)를 없애준다.
  • 빈도수/카운팅 문제가 나오면 Map + merge가 정석.

15JDBC 패턴

순수 JDBC. 레거시 유지보수 현장에 아직 있고, PreparedStatement/SQL Injection은 면접 단골.

면접 포인트
  • Statement vs PreparedStatement: 문자열을 +로 이어 붙이는 Statement는 SQL Injection에 취약. ? 바인딩하는 PreparedStatement가 안전 + 파싱 캐싱으로 성능도 유리.
  • JDBC 순서: 연결 → 쿼리 준비 → 파라미터 바인딩 → 실행 → 결과 처리 → 자원 해제. 자원 해제는 try-with-resources로.
  • executeQuery(SELECT, ResultSet) vs executeUpdate(DML, int 행수).

16Spring 형태

Controller-Service-Repository 3계층 + 생성자 주입. 스프링 프로젝트의 기본 뼈대.

면접 포인트
  • 필드 주입(@Autowired 필드) vs 생성자 주입: 생성자 주입이 권장. 이유 — ①불변(final) 보장 ②테스트 용이(new로 주입) ③순환참조를 기동 시점에 발견.
  • DI(의존성 주입): 객체 생성·연결을 스프링 컨테이너(IoC)가 담당 → 결합도 낮춤.
  • @Transactional: 메서드 내 예외 발생 시 롤백(기본은 RuntimeException). 조회만 하면 readOnly=true로 성능 최적화.
  • @Controller(뷰 반환) vs @RestController(=@Controller+@ResponseBody, JSON 반환).
