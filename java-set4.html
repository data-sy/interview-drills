Q31객관식 · checked vs unchecked 예외

checked 예외와 unchecked 예외의 구분으로 옳은 것은?

  1. unchecked는 RuntimeException 계열로 컴파일러가 처리를 강제하지 않는다
  2. checked는 컴파일러가 검사하지 않는다
  3. NullPointerException은 checked 예외다
  4. 모든 예외는 반드시 try-catch로 처리해야 한다
정답 확인
① unchecked는 RuntimeException 계열로 컴파일러가 처리를 강제하지 않는다
RuntimeException 하위(NPE, IllegalArgumentException 등)는 unchecked라 처리 강제가 없고, IOException 같은 checked는 try-catch나 throws가 필수다. 예외 설계는 장애 로깅·복구 전략과 직결된다.
Q32객관식 · try-catch-finally

try에서 예외가 발생하고 catch가 이를 처리할 때 finally 블록은?

  1. 실행되지 않는다
  2. catch보다 먼저 실행된다
  3. catch 실행 후 (return이 있어도) 실행된다
  4. 예외가 없을 때만 실행된다
정답 확인
③ catch 실행 후 (return이 있어도) 실행된다
finally는 예외 유무나 return과 무관하게 항상 실행된다(System.exit 등 예외 제외). DB 커넥션·파일 스트림 해제를 finally에 두어 자원 누수를 막는다.
Q33객관식 · throw vs throws

throw와 throws의 차이로 옳은 것은?

  1. throw는 메서드 선언부에 예외를 명시하고 throws는 예외를 실제로 던진다
  2. throw는 예외 객체를 실제로 발생시키고, throws는 메서드가 던질 수 있는 예외를 선언한다
  3. 둘은 완전히 같다
  4. throws는 예외를 잡는(catch) 키워드다
정답 확인
② throw는 예외 객체를 실제로 발생시키고, throws는 메서드가 던질 수 있는 예외를 선언한다
throw는 문장으로 예외를 발생시키고, throws는 선언으로 호출자에게 예외 처리를 위임한다는 통보다. 계층 간 예외 전파 규약을 명확히 해야 SI 공통 모듈에서 장애 원인 추적이 쉽다.
Q34객관식 · equals/hashCode 계약

equals()를 재정의할 때 hashCode()도 함께 재정의해야 하는 이유는?

  1. 재정의하지 않으면 컴파일 에러가 나기 때문
  2. 상속을 가능하게 하기 위해서
  3. 성능 향상만을 위해서
  4. equals가 true인 두 객체는 hashCode도 같아야 해시 기반 컬렉션이 올바로 동작하기 때문
정답 확인
④ equals가 true인 두 객체는 hashCode도 같아야 해시 기반 컬렉션이 올바로 동작하기 때문
이 계약을 어기면 HashMap/HashSet에서 같은 객체를 찾지 못하는 버그가 난다. 상품코드·계좌 같은 값 객체를 키로 쓸 때는 반드시 두 메서드를 쌍으로 재정의한다.
Q35객관식 · 오토박싱

Integer i = 10; 처럼 기본형이 대응하는 래퍼 객체로 자동 변환되는 것을 무엇이라 하는가?

  1. 오토박싱
  2. 언박싱
  3. 캐스팅
  4. 타입 추론
정답 확인
① 오토박싱
기본형에서 래퍼로의 자동 변환이 오토박싱, 반대는 언박싱이다. 편리하지만 반복 루프에서 불필요한 객체가 생성되고 null 언박싱 시 NPE 위험이 있어 대량 연산에는 기본형을 쓴다.
Q36객관식 · enum

자바 enum에 대한 설명으로 옳은 것은?

  1. 서로 다른 enum 상수는 == 비교가 불가능하다
  2. enum은 인터페이스를 구현할 수 없다
  3. 고정된 상수 집합을 타입 안전하게 표현하며 필드·메서드를 가질 수 있다
  4. enum 값은 런타임에 동적으로 추가된다
정답 확인
③ 고정된 상수 집합을 타입 안전하게 표현하며 필드·메서드를 가질 수 있다
enum은 타입 안전한 상수 열거로 switch·== 비교가 가능하고 필드/메서드도 둘 수 있다. 주문상태·회원등급 같은 유한 집합을 문자열 상수 대신 enum으로 두면 오타 버그가 사라진다.
Q37객관식 · 값/참조 전달

자바의 메서드 인자 전달 방식에 대한 설명으로 옳은 것은?

  1. 기본형은 값으로, 객체는 참조 자체를 전달해 호출자의 참조가 바뀔 수 있다
  2. 자바는 항상 값에 의한 전달이며 객체는 참조값(주소)의 복사본을 넘긴다
  3. 모든 인자는 참조로 전달된다
  4. 배열은 인자로 전달할 수 없다
정답 확인
② 자바는 항상 값에 의한 전달이며 객체는 참조값(주소)의 복사본을 넘긴다
자바는 항상 pass-by-value다. 객체는 참조값을 복사해 넘기므로 메서드 안에서 객체 내부 필드는 바꿀 수 있어도 호출자의 참조 자체를 다른 객체로 바꾸지는 못한다. 이 오해가 "왜 값이 안 바뀌지" 디버깅 시간을 잡아먹는다.
Q38코드읽기 · finally와 return

다음 메서드의 반환값을 쓰시오.

int f() {
    try { return 1; }
    finally { return 2; }
}
모범답안 확인
모범답안 · 2
finally의 return이 try의 return을 덮어써 최종적으로 2가 반환된다. finally에 return을 두면 예외까지 삼켜 장애를 숨기므로 실무에서는 금기로 다룬다.
Q39단답 · try-with-resources

try-with-resources 구문의 목적을 한 줄로 쓰시오.

모범답안 확인
모범답안
try 괄호 안에 선언한 (AutoCloseable) 자원을 블록이 끝날 때 자동으로 close 해주어 자원 누수를 방지한다.

finally에서 수동으로 close하던 상용구를 대체한다. DB 커넥션·스트림 누수는 운영 서버 다운의 흔한 원인이라 이 구문 사용이 권장된다.
Q40서술형 · 예외 처리 전략

실무에서 checked 예외를 만났을 때의 처리 원칙을 "복구 가능/불가능" 관점에서 설명하시오.

모범답안 확인
모범답안
현재 계층에서 복구가 가능하면 try-catch로 처리(재시도·대체값 반환)하고, 복구가 불가능하면 상위로 throws 전파하거나 의미 있는 런타임 예외로 감싸(wrapping) 던진다. 어느 경우든 예외를 빈 catch로 삼키지 않고 반드시 로깅한다.

채점 포인트는 복구 여부에 따른 처리/전파 결정, 로깅, 예외 무시 금지다. 고객 장애 대응에서 원인 추적 로그의 유무가 처리 시간을 좌우한다.

이 세트 우선복습 3줄

  1. checked vs unchecked — RuntimeException 계열은 unchecked라 처리 강제가 없고, IOException 같은 checked는 try-catch/throws가 필수. NPE는 unchecked라는 점이 함정.
  2. finally는 무조건 실행 — 예외·return과 무관하게 항상 돈다. 단 finally에 return을 두면 try의 반환값을 덮어쓰고 예외까지 삼켜 장애를 숨기므로 금기.
  3. equals/hashCode 쌍 재정의 — equals가 true면 hashCode도 같아야 한다. 하나만 재정의하면 HashMap/HashSet에서 값 객체를 못 찾는 버그가 난다.
