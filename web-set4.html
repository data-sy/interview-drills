Q31객관식 · HTTP 메서드(safe/DELETE)

HTTP 메서드에 대한 설명으로 옳은 것은?

  1. DELETE는 지정한 자원을 삭제 요청하는 메서드다
  2. GET은 서버 자원을 변경하는 안전하지 않은 메서드다
  3. DELETE는 조회 전용이라 서버 상태를 바꾸지 않는다
  4. 모든 메서드는 본문(body) 없이 사용해야 한다
정답 확인
① DELETE는 지정한 자원을 삭제 요청하는 메서드다
DELETE는 자원 삭제를 요청하는 메서드이고, GET은 자원을 변경하지 않는 '안전한(safe)' 메서드다. 조회 링크(GET)로 삭제 기능을 만들면 크롤러·프리페치가 자원을 지울 수 있어, SI에서 삭제는 반드시 DELETE/POST로 구현해야 하는 이유가 된다.
Q32객관식 · 상태코드 404 vs 500

404와 500 상태코드의 의미로 옳은 것은?

  1. 404는 서버 내부 오류, 500은 페이지 없음이다
  2. 404는 요청한 자원을 찾을 수 없음, 500은 서버 내부 오류다
  3. 둘 다 정상 응답이다
  4. 404는 인증 실패, 500은 리다이렉트다
정답 확인
② 404는 요청한 자원을 찾을 수 없음, 500은 서버 내부 오류다
404 Not Found는 요청 URL의 자원이 없을 때, 500 Internal Server Error는 서버 로직·예외로 처리에 실패했을 때다. 고객사 오류 신고 시 404면 경로·라우팅을, 500이면 애플리케이션 로그·예외 스택을 봐야 하므로 1차 원인 분류의 핵심 구분이다.
Q33객관식 · 응답 메시지 구조·헤더

HTTP 응답 메시지의 첫 줄(상태 라인)에 포함되는 정보로 옳은 것은?

  1. 요청 메서드와 요청 URI
  2. 클라이언트의 쿠키 값
  3. HTTP 버전, 상태 코드, 상태 메시지(예: HTTP/1.1 200 OK)
  4. 본문의 바이트 수만 표시된다
정답 확인
③ HTTP 버전, 상태 코드, 상태 메시지(예: HTTP/1.1 200 OK)
응답의 상태 라인은 'HTTP버전 상태코드 상태메시지'(예: HTTP/1.1 200 OK)로 구성되고, 이어서 헤더와 본문이 온다. 메서드·URI는 요청 라인의 정보다. API 응답을 디버깅할 때 상태 라인만 봐도 성공/실패를 즉시 판별할 수 있어 SI 인터페이스 점검의 첫 단계다.
Q34객관식 · CORS Preflight(OPTIONS)

CORS에서 Preflight(사전 요청)에 대한 설명으로 옳은 것은?

  1. 브라우저가 실제 요청 전에 OPTIONS 메서드로 서버에 허용 여부를 미리 확인하는 요청이다
  2. 서버가 클라이언트에 먼저 보내는 로그인 요청이다
  3. GET 요청 시 항상 두 번 전송되는 본 요청이다
  4. 쿠키를 암호화하는 절차다
정답 확인
① 브라우저가 실제 요청 전에 OPTIONS 메서드로 서버에 허용 여부를 미리 확인하는 요청이다
특정 조건(커스텀 헤더, PUT/DELETE 등)의 교차 출처 요청 전에 브라우저는 OPTIONS로 Preflight를 보내 서버가 해당 메서드·헤더를 허용하는지 확인한다. SI에서 프론트-API 분리 구조의 CORS 오류가 실제 요청이 아닌 이 OPTIONS 응답 설정 누락에서 나는 경우가 많다.
Q35객관식 · JWT 구조

JWT(JSON Web Token)의 구조로 옳은 것은?

  1. Header.Payload.Signature 세 부분이 점(.)으로 구분된다
  2. ID.Password 두 부분으로 구성된다
  3. 하나의 암호화된 XML 문서다
  4. 쿠키와 세션ID의 결합이다
정답 확인
① Header.Payload.Signature 세 부분이 점(.)으로 구분된다
JWT는 헤더(알고리즘 정보)·페이로드(클레임/사용자 정보)·서명(위변조 검증) 세 부분을 점으로 이은 토큰이다. 페이로드는 Base64 인코딩일 뿐 암호화가 아니어서 민감정보를 넣으면 안 된다는 점이, SI에서 토큰 기반 인증을 설계할 때 자주 놓치는 보안 포인트다.
Q36객관식 · Spring IoC 컨테이너·Bean

스프링에서 'Bean'에 대한 설명으로 옳은 것은?

  1. 개발자가 new로 직접 생성해 관리하는 임시 객체다
  2. 스프링 IoC 컨테이너가 생성·관리하는 객체로, 기본 스코프는 싱글톤이다
  3. 요청마다 무조건 새로 생성되는 객체다
  4. DB 테이블과 1:1로 매핑되는 엔티티만 가리킨다
정답 확인
② 스프링 IoC 컨테이너가 생성·관리하는 객체로, 기본 스코프는 싱글톤이다
Bean은 스프링 컨테이너가 생성·의존성 주입·소멸까지 관리하는 객체이며, 별도 지정이 없으면 싱글톤으로 한 개 인스턴스를 공유한다. 싱글톤 Bean에 상태를 저장하면 동시성 문제가 생기므로, SI 서비스 계층을 무상태로 짜야 하는 이유가 여기 있다.
Q37객관식 · HTML 시맨틱 태그

시맨틱(semantic) HTML 태그를 사용하는 이유로 가장 적절한 것은?

  1. div만 쓰는 것보다 요소의 의미를 명확히 해 접근성·SEO·유지보수에 유리하다
  2. 화면 색상을 자동으로 지정해 준다
  3. 자바스크립트 실행 속도를 높인다
  4. 서버 부하를 직접 줄여 준다
정답 확인
① div만 쓰는 것보다 요소의 의미를 명확히 해 접근성·SEO·유지보수에 유리하다
header·nav·main·article·footer 같은 시맨틱 태그는 문서 구조의 의미를 명시해 검색엔진·스크린리더가 내용을 잘 해석하게 하고, 코드 가독성도 높인다. 공공·금융 SI에서 웹 접근성 준수가 요구되는 경우가 많아 시맨틱 마크업이 실무 기준이 된다.
Q38서술형 · AJAX

AJAX가 무엇인지 설명하고, 동기 방식 대비 어떤 장점이 있는지 서술하시오.

모범답안 확인
모범답안
AJAX(Asynchronous JavaScript and XML)는 자바스크립트로 서버와 비동기 통신해, 페이지 전체를 새로고침하지 않고 필요한 데이터만 받아 화면 일부만 갱신하는 기법이다. 동기 방식은 응답이 올 때까지 화면이 멈추고 전체를 다시 그리지만, AJAX는 사용자가 다른 작업을 계속하며 부분 갱신을 받아 반응성과 체감 속도가 좋다. 현재는 주로 JSON 데이터를 주고받는다.

채점 포인트는 비동기·부분 갱신, 전체 리로드 없음, 반응성 향상이다. SI 조회·통계 화면에서 필터를 바꿀 때마다 전체 페이지를 다시 그리면 느리므로, AJAX로 데이터 영역만 갱신해 사용성을 높이는 것이 일반적이다.
Q39서술형 · 인증/인가·JWT

JWT를 사용한 토큰 기반 인증이 세션 방식과 비교해 어떻게 동작하며 어떤 장점이 있는지 서술하시오.

모범답안 확인
모범답안
로그인 성공 시 서버가 사용자 정보를 담고 서명한 JWT를 발급하고, 클라이언트는 이후 요청 헤더(주로 Authorization: Bearer)에 토큰을 실어 보낸다. 서버는 서명을 검증해 사용자를 식별하므로 세션을 서버에 저장할 필요가 없다(무상태). 덕분에 서버 확장(스케일아웃)과 여러 서버·서비스 간 인증 공유가 쉽고, 별도 세션 저장소 부담이 줄어든다.

채점 포인트는 토큰 발급·헤더 전달·서명 검증 흐름, 무상태로 서버 확장 용이라는 장점이다. 다만 토큰 탈취·만료 관리 이슈가 있어, SI에서 만료시간·리프레시 토큰 설계를 함께 고려해야 한다는 점도 가점 요소다.
Q40서술형 · 필터/인터셉터(공통처리)

웹 애플리케이션에서 필터(또는 인터셉터)를 사용해 처리하기 적합한 '공통 관심사'의 예를 들고, 이를 개별 로직에 넣지 않고 필터에서 처리하는 이점을 서술하시오.

모범답안 확인
모범답안
인증·권한 체크, 로그인 여부 확인, 요청/응답 로깅, 문자 인코딩 설정, 공통 예외/헤더 처리 등이 필터·인터셉터로 처리하기 적합한 공통 관심사다. 이를 각 컨트롤러마다 중복 작성하지 않고 앞단에서 일괄 처리하면, 코드 중복이 줄고 정책이 한 곳에 모여 변경·유지보수가 쉬워지며 누락을 방지할 수 있다.

채점 포인트는 공통 관심사 예시, 중복 제거·일괄 처리·유지보수 용이 이점이다. 화면·기능이 수십 개인 SI 시스템에서 인증·로깅을 공통 계층으로 빼면 요구 변경 시 한 곳만 고치면 되어 유지보수 비용이 크게 준다.

이 세트 우선복습 3줄

  1. CORS Preflight(OPTIONS) — 커스텀 헤더·PUT/DELETE 등의 교차 출처 요청 전에 브라우저가 OPTIONS로 사전 확인한다. CORS 오류는 본 요청이 아닌 이 OPTIONS 응답 설정 누락에서 자주 난다.
  2. JWT 페이로드는 암호화가 아니다 — Header.Payload.Signature 3부분 중 페이로드는 Base64 인코딩일 뿐이라 누구나 디코딩할 수 있다. 민감정보를 넣으면 안 된다.
  3. Bean 기본 스코프 = 싱글톤 — 스프링 컨테이너가 관리하는 Bean은 기본적으로 한 인스턴스를 공유한다. 싱글톤 Bean에 상태를 두면 동시성 문제가 생기므로 서비스 계층은 무상태로 짠다.
