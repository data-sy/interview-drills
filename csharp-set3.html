Q21객관식 · LINQ vs Stream

자바 Stream API와 C# LINQ 대비로 옳은 것은?

  1. LINQ는 자바 Stream과 달리 SQL 데이터베이스에서만 동작하고 메모리 컬렉션엔 못 쓴다
  2. 자바 stream().filter().map()에 대응해 C#은 LINQ의 Where().Select()(또는 쿼리식)로 컬렉션을 선언적으로 처리한다
  3. C#에는 스트림형 컬렉션 처리가 없어 항상 for문으로만 다룬다
  4. 자바 Stream은 지연 실행이지만 LINQ는 반드시 즉시 실행된다
정답 확인
② 자바 stream().filter().map() ↔ C# LINQ의 Where().Select()
자바 Stream의 filter/map/collect는 C# LINQ의 Where/Select/ToList(또는 from ... where ... select 쿼리식)에 대응한다. 둘 다 지연 실행 성향을 갖는다. 고객사 데이터 집계·변환 로직을 for문 대신 LINQ로 쓰면 의도가 드러나 유지보수가 쉬워진다.
Q22객관식 · 델리게이트

메서드를 값처럼 전달하는 기능의 자바↔C# 대비로 옳은 것은?

  1. 자바는 함수형 인터페이스(예: Function, Runnable)+람다로, C#은 델리게이트(Func<>, Action<>)로 메서드를 참조·전달한다
  2. C#에는 메서드를 전달하는 수단이 없어 항상 인터페이스로만 콜백을 구현한다
  3. 델리게이트는 자바 키워드이고 C#은 함수형 인터페이스만 쓴다
  4. 자바 람다는 델리게이트 타입이라 C#과 문법이 완전히 같다
정답 확인
① 자바 함수형 인터페이스+람다 ↔ C# 델리게이트(Func/Action)
자바는 함수형 인터페이스(Function/Consumer/Runnable)에 람다를 대입해 메서드를 값처럼 다루고, C#은 델리게이트 타입(대표적으로 Func<>, Action<>)으로 같은 일을 한다. 개념은 "메서드를 담는 타입 안전한 참조"로 같다. 이벤트 콜백·전략 주입에 쓰여 결제·검증 로직을 유연하게 갈아끼운다.
Q23객관식 · 이벤트 event

C#의 event 키워드에 대한 자바 대비 설명으로 옳은 것은?

  1. 자바에는 이벤트/리스너 개념이 전혀 없다
  2. C#의 event는 외부에서 자유롭게 호출(invoke)할 수 있어 캡슐화가 없다
  3. event는 자바 키워드이고 C#은 Observer를 직접 구현해야만 한다
  4. 자바는 리스너 인터페이스+addXxxListener 패턴으로 이벤트를 구현하지만, C#은 delegate 기반 event 키워드로 언어 차원에서 발행/구독을 지원한다
정답 확인
④ 자바 리스너+addXxxListener ↔ C# delegate 기반 event 키워드
자바는 리스너 인터페이스 + addActionListener 같은 관례로 옵저버 패턴을 구현하지만, C#은 delegate 위에 event 키워드로 발행/구독을 언어가 직접 지원한다. event는 외부에서 += / -=만 되고 직접 호출은 선언 클래스 내부로 제한돼 캡슐화된다. UI/알림·상태 변경 통지 같은 이벤트 주도 로직에서 결합도를 낮춘다.
Q24객관식 · using/IDisposable

자원 자동 해제의 자바↔C# 대비로 옳은 것은?

  1. C#은 자원 해제를 GC에만 맡겨 명시적 해제 구문이 없다
  2. IDisposable은 자바 인터페이스이고 C#은 AutoCloseable을 구현한다
  3. 자바 try-with-resources(AutoCloseable)에 대응해 C#은 using 문(IDisposable)으로 블록 종료 시 Dispose를 자동 호출한다
  4. using은 자바 키워드이고 C#은 try-with-resources를 그대로 쓴다
정답 확인
③ 자바 try-with-resources(AutoCloseable) ↔ C# using 문(IDisposable)
자바 try-with-resources는 AutoCloseable.close()를, C# using 문은 IDisposable.Dispose()를 블록 종료 시 자동 호출한다. 둘 다 예외가 나도 확실히 해제한다. DB 커넥션·파일·스트림을 다루는 SI 코드에서 using을 빼먹으면 커넥션 누수로 운영 장애가 난다.
Q25객관식 · 제네릭 소거 vs reified

제네릭의 런타임 타입 정보 처리에서 자바↔C# 차이로 옳은 것은?

  1. C#도 자바처럼 타입 소거라 런타임에 List<int>와 List<string>을 구분 못 한다
  2. 자바는 타입 소거(erasure)로 런타임에 타입 인자가 지워지지만, C#은 제네릭 타입 정보를 런타임까지 보존(reified)한다
  3. 자바는 타입을 보존하고 C#이 소거하는 정반대 구조다
  4. 두 언어 모두 제네릭이 순수 문법 설탕이라 런타임엔 존재하지 않는다
정답 확인
② 자바 타입 소거(erasure) ↔ C# reified(런타임 보존)
자바 제네릭은 타입 소거라 런타임에 List<String>이 그냥 List가 되지만, C#은 reified 제네릭이라 typeof(T)로 런타임 타입을 알 수 있고 new T[]도 가능하다. 그래서 C#은 자바에서 못 하는 제네릭 리플렉션·인스턴스화가 된다. 공용 프레임워크 코드에서 이 차이를 알아야 이관 시 리플렉션 로직이 깨지지 않는다.
Q26객관식 · boxing/unboxing

값 타입과 객체 간 변환의 자바↔C# 대비로 옳은 것은?

  1. 자바 오토박싱(int↔Integer)에 대응해 C#은 값 형식↔object 변환을 박싱/언박싱이라 부르며, 둘 다 힙 할당 비용이 든다
  2. C#에는 박싱 개념이 없어 값 형식과 object 간 변환이 무비용이다
  3. 자바 오토박싱은 힙을 안 쓰지만 C# 박싱만 힙을 쓴다
  4. 박싱은 자바 전용 용어이고 C#은 캐스팅이라 부르며 비용이 없다
정답 확인
① 자바 오토박싱(int↔Integer) ↔ C# 박싱/언박싱, 둘 다 힙 비용
자바가 int↔Integer를 오토박싱하듯, C#은 값 형식을 object(또는 인터페이스)로 담을 때 박싱, 되꺼낼 때 언박싱하며 힙 할당·복사 비용이 든다. 둘 다 반복문 안 대량 박싱은 성능 저하다. 그래서 두 언어 모두 제네릭 컬렉션(List<T>)으로 박싱을 피한다. 대량 통계 처리에서 박싱 남발이 GC 압박의 흔한 원인이다.
Q27객관식 · StringBuilder

문자열 누적 조작 도구의 자바↔C# 대비로 옳은 것은?

  1. C#에는 StringBuilder가 없어 항상 string + string으로만 연결한다
  2. C# StringBuilder는 불변이라 자바 String과 동일하다
  3. 자바 StringBuffer가 C#의 기본 string 타입과 같은 것이다
  4. 자바 StringBuilder(비동기)/StringBuffer(동기)에 대응해 C#은 StringBuilder로 가변 문자열 누적을 처리한다
정답 확인
④ 자바 StringBuilder/StringBuffer ↔ C# StringBuilder(가변 누적)
자바는 단일 스레드용 StringBuilder와 동기화된 StringBuffer로 나뉘지만, C#은 StringBuilder 하나로 가변 문자열 누적을 처리한다(스레드 안전이 필요하면 별도 동기화). 반복 연결에서 불변 string 대신 이걸 써야 O(n) 재할당을 피한다. 대량 리포트/쿼리 문자열 생성 배치에서 성능 핵심 도구다.
Q28객관식 · equals/GetHashCode

객체 동등성 비교 재정의의 자바↔C# 대비로 옳은 것은?

  1. C#은 GetHashCode만 재정의하면 되고 Equals는 재정의할 필요가 없다
  2. C#은 == 연산자만 있으면 되고 Equals 개념 자체가 없다
  3. 자바는 equals()/hashCode()를 함께 재정의하고, C#은 Equals()/GetHashCode()를 함께 재정의한다(둘 다 쌍으로 맞춰야 함)
  4. 자바 equals와 C#의 == 는 항상 값 비교로 완전히 동일하게 동작한다
정답 확인
③ 자바 equals()/hashCode() ↔ C# Equals()/GetHashCode() (쌍으로 재정의)
자바 equals()/hashCode()에 대응하는 C#은 Equals()/GetHashCode()이며, 둘 다 논리적 동등성과 해시 기반 컬렉션(HashSet/Dictionary)의 정합을 위해 쌍으로 재정의해야 한다. C#은 추가로 == 연산자 오버로드도 고려한다. 도메인 키(고객ID 등)로 중복 제거·조회할 때 이 쌍을 어기면 컬렉션이 오작동한다.
Q29코드읽기 · LINQ 결과

다음 C# LINQ 코드의 출력은? 자바 Stream filter/map/collect와 대비하며 읽어라.

int[] nums = { 1, 2, 3, 4, 5, 6 };
var result = nums.Where(n => n % 2 == 0)
                 .Select(n => n * 10)
                 .ToList();
Console.WriteLine(string.Join(",", result));
모범답안 확인
모범답안 · 20,40,60
자바로 치면 stream().filter(n->n%2==0).map(n->n*10).collect(...)와 같다. Where로 짝수(2,4,6)만 남기고 Select로 각각 10배 해 20,40,60을 만든다. ToList()에서 실제 실행(구체화)된다. 데이터 필터+변환 파이프라인을 선언적으로 표현하는 실무 전형이다.
Q30단답 · null 병합 ??

자바에서 name != null ? name : "익명"처럼 쓰는 null 대체를, C#의 null 병합 연산자로 간결하게 쓰시오.

모범답안 확인
모범답안 · name ?? "익명"
name ?? "익명" — 자바는 삼항 연산자/Optional.orElse, C#은 ?? 연산자를 쓴다.

자바는 삼항 연산자나 Optional.orElse("익명")로 null 대체를 하고, C#은 name ?? "익명"으로 왼쪽이 null이면 오른쪽을 반환한다. 대입 단축형 ??=도 있다. DB나 외부 연동에서 들어온 null 값을 기본값으로 방어할 때 코드가 짧고 안전해진다.

이 세트 우선복습 3줄

  1. 제네릭 소거 vs reified — 자바는 타입 소거라 런타임에 List<T>의 T가 지워지지만, C#은 reified라 typeof(T)·new T[]가 된다. 리플렉션 이관의 함정.
  2. using / IDisposable — 자바 try-with-resources(AutoCloseable) ↔ C# using 문(IDisposable.Dispose). 빼먹으면 DB 커넥션 누수로 장애.
  3. Equals/GetHashCode는 쌍 — 자바 equals()/hashCode()처럼 C#도 Equals()/GetHashCode()를 함께 재정의해야 HashSet/Dictionary가 정상 동작한다.
