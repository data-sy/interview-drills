Q41객관식 · HTTP 메서드-CRUD 매핑

REST에서 CRUD 작업과 HTTP 메서드의 매핑으로 옳은 것은?

  1. 조회-GET, 생성-POST, 수정-PUT/PATCH, 삭제-DELETE
  2. 조회-POST, 생성-GET, 수정-DELETE, 삭제-PUT
  3. 조회-DELETE, 생성-PUT, 수정-GET, 삭제-POST
  4. 모든 작업을 GET 하나로 처리한다
정답 확인
① 조회-GET, 생성-POST, 수정-PUT/PATCH, 삭제-DELETE
조회는 GET, 생성은 POST, 수정은 PUT/PATCH, 삭제는 DELETE로 매핑하는 것이 REST 관례다. 이 규칙을 지키면 URL만 봐도 동작을 유추할 수 있어, 여러 팀·외부사가 참여하는 SI API 설계·문서화에서 소통 비용을 줄인다.
Q42객관식 · 상태코드 200/201/204

다음 상태코드의 의미로 옳지 않은 것은?

  1. 200 OK — 요청 성공(응답 본문 있음)
  2. 201 Created — 리소스 생성 성공
  3. 204 No Content — 성공했지만 응답 본문이 없음
  4. 200 OK — 리소스가 삭제되어 더 이상 없음을 의미
정답 확인
④ 200 OK — 리소스가 삭제되어 더 이상 없음을 의미
200은 일반 성공 응답으로 '삭제되어 없음'을 뜻하지 않는다. 자원 생성 성공은 201, 본문 없는 성공(삭제·수정 후 등)은 204로 구분하는 것이 REST 관례다. SI API에서 등록 후 201·삭제 후 204를 정확히 내려주면 클라이언트가 후속 처리를 명확히 분기할 수 있다.
Q43객관식 · 무상태성(대응 수단)

HTTP가 무상태(Stateless)이기 때문에 로그인 상태를 유지하기 위해 사용하는 수단이 아닌 것은?

  1. 세션(서버 저장 + 세션ID 쿠키)
  2. 쿠키
  3. JWT 같은 토큰
  4. HTTP 상태코드 200
정답 확인
④ HTTP 상태코드 200
상태코드 200은 요청 처리 결과를 알리는 값일 뿐 상태 유지 수단이 아니다. 무상태 특성상 요청마다 사용자를 식별하려면 세션·쿠키·토큰이 필요하다. SI에서 로그인 이후 화면들이 사용자를 계속 인식하게 하려면 이 수단 중 하나를 반드시 설계해야 한다.
Q44객관식 · REST API 응답 설계

REST API에서 '존재하지 않는 회원 ID로 GET /users/999를 조회'했을 때 가장 적절한 응답은?

  1. 200 OK와 빈 본문
  2. 404 Not Found
  3. 500 Internal Server Error
  4. 301 Moved Permanently
정답 확인
② 404 Not Found
요청한 자원이 없으면 404 Not Found가 의미상 맞다. 없는 데이터를 200으로 내리면 클라이언트가 성공으로 오해하고, 500은 서버 오류로 왜곡된다. SI 연계 API에서 상태코드를 상황에 맞게 정확히 주는 것이 상대 시스템의 오류 처리·재시도 로직 품질을 좌우한다.
Q45코드읽기 · JS/DOM 조작

다음 자바스크립트 코드의 동작으로 옳은 것은?

document.getElementById("msg").textContent = "완료";
  1. id가 "msg"인 요소를 새로 생성한다
  2. 서버에 "완료"라는 데이터를 전송한다
  3. "msg"라는 클래스의 모든 요소를 삭제한다
  4. id가 "msg"인 요소의 텍스트 내용을 "완료"로 바꾼다
정답 확인
④ id가 "msg"인 요소의 텍스트 내용을 "완료"로 바꾼다
getElementById로 해당 id의 DOM 요소를 찾아 textContent에 값을 대입하면 그 요소의 텍스트가 교체된다. AJAX 응답 후 결과 메시지 표시처럼 화면을 동적으로 갱신하는 기본 패턴으로, SI 화면의 유효성 안내·상태 표시 구현에 흔히 쓰인다.
Q46객관식 · 웹서버 vs WAS(정적/동적)

일반적인 웹 시스템 구성에서 요청 처리에 대한 설명으로 옳은 것은?

  1. 이미지·CSS 같은 정적 자원은 WAS만 처리할 수 있다
  2. 정적 자원은 웹서버가 바로 응답하고, JSP·서블릿 같은 동적 요청은 WAS로 전달해 처리한다
  3. 웹서버는 DB에 직접 접속해 SQL을 실행한다
  4. WAS는 정적 자원만 담당한다
정답 확인
② 정적 자원은 웹서버가 바로 응답하고, JSP·서블릿 같은 동적 요청은 WAS로 전달해 처리한다
웹서버(Apache·Nginx)는 정적 자원을 빠르게 응답하고, 동적 처리(로직·DB 연동)가 필요한 요청만 뒤단 WAS(Tomcat 등)로 넘긴다. 이 분리로 부하 분산·보안·확장이 쉬워져, SI 구축에서 웹서버-WAS 2계층 구성이 표준으로 쓰인다.
Q47객관식 · 동일출처정책(Origin 판정)

현재 페이지가 'https://shop.com:443'일 때, 동일 출처(Same-Origin)로 간주되는 것은?

  1. http://shop.com (프로토콜 다름)
  2. https://api.shop.com (도메인 다름)
  3. https://shop.com:8080 (포트 다름)
  4. https://shop.com/products (경로만 다름)
정답 확인
④ https://shop.com/products (경로만 다름)
출처(origin)는 프로토콜·도메인·포트 세 가지가 모두 같아야 동일 출처이고, 경로(path)는 판정에 포함되지 않는다. 따라서 경로만 다른 ④가 동일 출처다. 프론트와 API의 서브도메인·포트가 달라 CORS가 걸리는 SI 상황을 진단할 때 이 origin 판정 기준이 핵심이다.
Q48서술형 · 상태코드(실무 활용)

REST API를 설계할 때 HTTP 상태코드를 상황에 맞게 구분해 반환하는 것이 왜 중요한지, 대표 코드 예를 들어 설명하시오.

모범답안 확인
모범답안
상태코드는 클라이언트가 응답을 해석하고 후속 처리를 분기하는 기준이므로, 상황에 맞게 정확히 줘야 한다. 예: 성공 200, 생성 201, 본문 없는 성공 204, 잘못된 요청 400, 미인증 401, 권한 없음 403, 자원 없음 404, 서버 오류 500. 모든 응답을 200으로 통일하고 본문 안에 성공/실패를 담으면 클라이언트가 정상/오류를 표준 방식으로 구분하지 못하고 오류 처리·모니터링이 어려워진다.

채점 포인트는 코드가 클라이언트 분기·오류 처리의 기준이라는 점과 대표 코드 구분이다. SI에서 외부사·모바일과 연계할 때 상태코드를 표준대로 내려야 상대 시스템의 재시도·알림·로깅이 제대로 동작하고 장애 원인 추적도 빨라진다.
Q49서술형 · HTML/CSS/JS·DOM(렌더링)

브라우저가 HTML·CSS·JS로 화면을 만드는 과정에서 HTML, CSS, 자바스크립트(DOM 조작)가 각각 어떤 역할을 하는지 설명하시오.

모범답안 확인
모범답안
브라우저는 HTML을 파싱해 문서 구조인 DOM 트리를 만들고, CSS를 해석해 스타일 정보를 결합한 뒤 화면에 그린다(렌더링). HTML은 문서의 구조와 내용을, CSS는 색·크기·배치 등 표현을 담당한다. 자바스크립트는 이 DOM에 접근해 요소를 추가·수정·삭제하거나 이벤트에 반응해 화면을 동적으로 바꾼다. 즉 HTML=구조, CSS=표현, JS=동작(DOM 조작)으로 관심사가 나뉜다.

채점 포인트는 HTML→DOM 생성, CSS=표현, JS=DOM 조작으로 동적 변경이라는 역할 분담이다. 이 흐름을 이해해야 SI 화면에서 초기 렌더링과 이후 동적 갱신(유효성 검사·AJAX 반영) 문제를 구분해 디버깅할 수 있다.
Q50서술형 · Spring 계층 어노테이션

스프링에서 @Controller, @Service, @Repository 어노테이션이 각각 어떤 계층·역할을 담당하는지 설명하시오.

모범답안 확인
모범답안
세 어노테이션은 모두 스프링이 관리하는 Bean으로 등록시키되 계층별 역할을 명시한다. @Controller는 프레젠테이션 계층에서 사용자 요청을 받아 처리 흐름을 제어하고 응답(화면·데이터)을 반환한다. @Service는 비즈니스 로직·트랜잭션을 담당하는 서비스 계층이다. @Repository는 DB 접근(DAO) 계층으로, 데이터 접근 예외를 스프링 예외로 변환하는 이점도 있다. 이렇게 계층을 나눠 역할과 책임을 분리한다.

채점 포인트는 Controller(요청 처리)-Service(비즈니스 로직)-Repository(데이터 접근)의 3계층 분리와 각 역할이다. 계층이 명확히 나뉘어야 담당자가 바뀌어도 코드 위치를 예측할 수 있고 변경 파급이 줄어, 대규모·장기 유지보수형 SI 프로젝트에서 표준 구조로 쓰인다.

이 세트 우선복습 3줄

  1. 상태코드는 200 남발 금지 — 생성 201·본문 없는 성공 204·자원 없음 404·서버 오류 500을 상황대로 내려야 클라이언트가 성공/실패를 표준 방식으로 분기한다.
  2. 동일 출처(origin) 판정 — 프로토콜·도메인·포트 세 가지가 모두 같아야 동일 출처이며, 경로(path)는 판정에 포함되지 않는다. 포트만 달라도 CORS가 걸린다.
  3. 무상태 유지 수단 — 요청마다 사용자를 식별하려면 세션·쿠키·토큰(JWT)이 필요하다. 상태코드 200은 처리 결과일 뿐 상태 유지 수단이 아니다.
