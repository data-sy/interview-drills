Q1객관식 · HTTP 메서드(GET/POST)

GET과 POST 메서드의 차이에 대한 설명으로 옳은 것은?

  1. GET은 요청 데이터를 본문(body)에, POST는 URL 쿼리스트링에 담는다
  2. GET은 본문 크기 제한이 없고 POST는 제한이 있다
  3. GET은 주로 조회, POST는 주로 생성/제출에 쓰며 GET 파라미터는 URL에 노출된다
  4. POST는 브라우저 캐시·북마크가 GET보다 쉽다
정답 확인
③ GET은 주로 조회, POST는 주로 생성/제출에 쓰며 GET 파라미터는 URL에 노출된다
GET은 리소스 조회용으로 파라미터가 URL 쿼리스트링에 붙어 노출·캐시·북마크가 되고, POST는 데이터를 본문에 담아 생성/변경 요청에 쓴다. SI에서 검색·조회 화면은 GET, 등록/결재 폼 전송은 POST로 설계해야 URL 노출과 재전송 문제를 줄인다.
Q2객관식 · 상태코드 2xx~5xx 대분류

HTTP 상태코드 대분류의 의미를 잘못 짝지은 것은?

  1. 2xx — 요청 성공
  2. 3xx — 리다이렉션(추가 동작 필요)
  3. 4xx — 클라이언트 오류
  4. 5xx — 클라이언트 요청 문법 오류
정답 확인
④ 5xx — 클라이언트 요청 문법 오류
5xx는 서버 측 오류이며, 클라이언트 요청 자체의 문제는 4xx다. 고객사 장애 문의를 받을 때 4xx면 요청·권한을, 5xx면 서버·DB 로그를 먼저 봐야 하므로 대분류 구분이 1차 분류의 기준이 된다.
Q3객관식 · 요청 메시지 구조

HTTP 요청 메시지의 구성 요소로 올바른 순서는?

  1. 요청 라인(메서드·URI·버전) → 헤더 → 본문
  2. 상태 라인 → 헤더 → 본문
  3. 헤더 → 요청 라인 → 본문
  4. 요청 라인 → 본문 → 헤더
정답 확인
① 요청 라인(메서드·URI·버전) → 헤더 → 본문
요청은 '요청 라인(메서드·경로·HTTP버전) → 헤더 → 빈 줄 → 본문' 순서이며, '상태 라인'은 응답 메시지의 첫 줄이다. 프록시·게이트웨이 로그를 분석하거나 API 연동 오류를 추적할 때 어느 라인·헤더가 문제인지 구조를 알아야 원인을 짚을 수 있다.
Q4객관식 · 쿠키 vs 세션(저장위치)

쿠키와 세션의 저장 위치에 대한 설명으로 옳은 것은?

  1. 쿠키는 서버 메모리, 세션은 클라이언트 브라우저에 저장된다
  2. 쿠키와 세션 모두 서버 DB에만 저장된다
  3. 쿠키는 클라이언트에, 세션 데이터는 서버에 저장되고 세션ID만 쿠키로 오간다
  4. 세션은 클라이언트 디스크에 영구 저장된다
정답 확인
③ 쿠키는 클라이언트에, 세션 데이터는 서버에 저장되고 세션ID만 쿠키로 오간다
쿠키는 브라우저에 저장되어 매 요청에 실려 가고, 세션 데이터는 서버가 보관하며 식별용 세션ID만 쿠키로 주고받는다. 민감정보(권한·로그인 상태)를 세션에 두는 이유가 여기 있고, SI 다중 서버 환경에선 세션 불일치 때문에 세션 클러스터링/스토리지 분리가 필요해진다.
Q5객관식 · MVC 패턴(역할)

웹 MVC 패턴에서 각 구성요소의 역할로 옳은 것은?

  1. Model은 화면 출력, View는 데이터 처리, Controller는 저장을 담당한다
  2. Model이 사용자 요청을 직접 받아 라우팅한다
  3. Controller는 화면을, View는 DB 접근을 담당한다
  4. Model은 데이터·비즈니스 로직, View는 화면 표현, Controller는 요청 처리·흐름 제어를 담당한다
정답 확인
④ Model은 데이터·비즈니스 로직, View는 화면 표현, Controller는 요청 처리·흐름 제어를 담당한다
Model은 데이터와 업무 로직, View는 사용자에게 보이는 화면, Controller는 요청을 받아 Model과 View를 연결하는 흐름 제어를 맡는다. 역할이 분리돼야 화면 변경이 로직에 영향을 주지 않아, 고객사 UI 수정 요청이 잦은 SI 유지보수에서 특히 유리하다.
Q6객관식 · REST/RESTful(URI 설계)

RESTful한 URI 설계로 가장 적절한 것은? (게시글 3번 조회)

  1. GET /getBoard?id=3
  2. POST /board/select/3
  3. GET /boards/3
  4. GET /boards/3/delete
정답 확인
③ GET /boards/3
REST는 URI에 자원을 명사로 표현하고 행위는 HTTP 메서드로 구분하므로, 조회는 'GET /boards/3'가 옳다. 동사(get/select/delete)를 URI에 넣지 않는다. SI에서 여러 팀·외부사와 API를 연동할 때 일관된 REST 규칙은 문서·소통 비용을 줄여준다.
Q7객관식 · HTML/CSS/JS 역할분담

웹 프론트엔드 3요소의 역할을 옳게 짝지은 것은?

  1. HTML은 스타일, CSS는 동작, JS는 구조
  2. HTML은 동작, CSS는 구조, JS는 스타일
  3. HTML은 구조·내용, CSS는 표현·스타일, JS는 동작·상호작용
  4. 셋 모두 화면 스타일만 담당한다
정답 확인
③ HTML은 구조·내용, CSS는 표현·스타일, JS는 동작·상호작용
HTML은 문서의 구조와 내용, CSS는 색·배치 등 표현, JS는 이벤트·동적 처리를 담당한다. 이 관심사 분리를 지켜야 퍼블리셔·개발자 협업과 화면 유지보수가 쉬워지며, SI 구축 프로젝트에서 마크업/스타일/스크립트 분업의 기준이 된다.
Q8서술형 · HTTP/HTTPS·SSL/TLS

HTTP와 HTTPS의 차이를 설명하고, HTTPS가 보안을 제공하기 위해 사용하는 SSL/TLS의 역할을 서술하시오.

모범답안 확인
모범답안
HTTP는 데이터를 평문으로 전송해 도청·변조·위장에 취약한 반면, HTTPS는 HTTP 통신에 SSL/TLS 암호화 계층을 더한 것이다. SSL/TLS는 (1) 서버 인증서로 상대 서버가 진짜인지 확인(인증), (2) 대칭키로 본문을 암호화(기밀성), (3) 데이터 변조 여부 검증(무결성)을 제공한다. 기본 포트는 HTTP 80, HTTPS 443이다.

채점 포인트는 평문 vs 암호화, SSL/TLS의 인증·기밀성·무결성 세 역할, 포트 차이다. 고객사 로그인·결제·개인정보 화면은 HTTPS 적용이 사실상 필수이고, 인증서 만료로 접속 장애가 나는 사례가 흔해 SI 운영에서 인증서 갱신 관리가 중요하다.
Q9서술형 · 무상태성

HTTP의 '무상태성(Stateless)'이 무엇인지 설명하고, 이로 인해 로그인 상태 유지를 위해 어떤 보완이 필요한지 서술하시오.

모범답안 확인
모범답안
무상태성이란 서버가 각 요청을 독립적으로 처리하며 이전 요청의 상태(누가 로그인했는지 등)를 기억하지 않는 특성이다. 그래서 요청마다 사용자를 식별할 수 없으므로, 쿠키·세션이나 토큰(JWT) 같은 별도 수단으로 상태를 유지·전달해야 한다. 무상태성 덕분에 서버 확장(스케일아웃)이 쉬워지는 장점도 있다.

채점 포인트는 '요청 간 상태를 기억하지 않음' 정의, 상태 유지 보완책(세션/토큰), 그리고 확장성 이점이다. SI에서 트래픽 증가 대비 서버를 늘릴 때 무상태 설계가 유리하지만, 세션을 서버 메모리에만 두면 서버 간 상태 공유 문제가 생긴다.
Q10서술형 · 웹서버 vs WAS

웹서버(Web Server)와 WAS(Web Application Server)의 차이를 역할 중심으로 설명하시오.

모범답안 확인
모범답안
웹서버는 HTML·이미지·CSS 같은 정적 콘텐츠를 클라이언트에 전달하는 서버(예: Apache, Nginx)다. WAS는 JSP·서블릿 등 동적 처리를 수행해 DB 연동·비즈니스 로직 결과를 만들어내는 서버(예: Tomcat, WebLogic)다. 실무에서는 웹서버가 앞단에서 정적 요청과 로드밸런싱을 맡고 동적 요청만 뒤단 WAS로 넘기는 구성이 일반적이다.

채점 포인트는 정적 vs 동적 처리 구분, 대표 제품 예시, 웹서버-WAS 분리 구성의 이유(부하 분산·보안)다. SI 구축 시 이 2계층 구조를 이해해야 정적 자원 캐싱, WAS 다중화 같은 성능·안정성 설계를 할 수 있다.

이 세트 우선복습 3줄

  1. 상태코드 대분류 — 4xx는 클라이언트 오류, 5xx는 서버 오류. 장애 1차 분류: 4xx면 요청·권한, 5xx면 서버·DB 로그부터.
  2. 쿠키 vs 세션 저장위치 — 쿠키는 클라이언트에, 세션 데이터는 서버에 저장되고 세션ID만 쿠키로 오간다.
  3. RESTful URI — 자원은 명사, 행위는 HTTP 메서드. 조회는 GET /boards/3지 /getBoard?id=3·/board/select/3가 아니다.
