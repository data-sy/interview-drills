VIEW·SELF JOIN·서브쿼리부터 다중정렬·CROSS JOIN까지

보기를 고른 뒤 채점하기를 누르면 정오답과 해설이 열립니다. 쿼리작성·단답은 직접 적어보고 모범답안과 비교해 스스로 채점하세요.

Q31객관식 · VIEW

뷰(VIEW)에 대한 설명으로 옳은 것은?

  1. 데이터를 물리적으로 복제해 별도 저장한다
  2. 인덱스를 대체하는 자료구조다
  3. 저장된 SELECT 문에 이름을 붙인 가상 테이블로, 조회 시 원본을 참조한다
  4. 뷰는 어떤 경우에도 갱신(UPDATE)할 수 없다
정답 확인
③ 저장된 SELECT 문에 이름을 붙인 가상 테이블로, 조회 시 원본을 참조한다
일반 뷰는 데이터를 저장하지 않고 실행 시 원본을 조회하는 논리적 창이다. SI에서 복잡한 조인·권한 제한 조회를 뷰로 캡슐화해 애플리케이션을 단순화하고, 민감 컬럼을 숨겨 보안 경계로도 활용한다.
Q32객관식 · SELF JOIN

셀프 조인(SELF JOIN)이 필요한 상황으로 가장 적절한 것은?

  1. 한 테이블 안에서 사원과 그 관리자처럼 같은 테이블의 행끼리 관계를 맺을 때
  2. 서로 다른 두 테이블을 합칠 때
  3. 중복 행을 제거할 때
  4. 집계 결과를 정렬할 때
정답 확인
① 한 테이블 안에서 사원과 그 관리자처럼 같은 테이블의 행끼리 관계를 맺을 때
셀프 조인은 동일 테이블에 서로 다른 별칭을 붙여 행 간 관계를 조회한다. 조직도(사원-상위관리자), 카테고리-상위카테고리 같은 계층 구조는 SI 인사/상품 시스템에 흔해, 별칭 구분이 핵심이다.
Q33객관식 · IN vs EXISTS

IN과 EXISTS(서브쿼리)에 대한 설명으로 옳은 것은?

  1. 둘은 문법만 다를 뿐 어떤 경우에도 성능이 완전히 동일하다
  2. IN은 서브쿼리를 쓸 수 없다
  3. EXISTS는 항상 IN보다 느리다
  4. EXISTS는 조건을 만족하는 행이 존재하는지만 확인해, 발견 즉시 참을 반환한다
정답 확인
④ EXISTS는 조건을 만족하는 행이 존재하는지만 확인해, 발견 즉시 참을 반환한다
EXISTS는 존재 여부만 보고 첫 매칭에서 멈추므로 상관 서브쿼리·대량 데이터에서 유리한 경우가 많다. SI 튜닝에서 "IN을 EXISTS로 바꿔 개선"하는 사례가 흔하며, 상황에 따라 실행계획을 보고 선택한다.
Q34객관식 · CHECK/DEFAULT

"값을 넣지 않으면 자동으로 특정 값이 저장되도록" 하는 제약조건은?

  1. CHECK
  2. DEFAULT
  3. UNIQUE
  4. FOREIGN KEY
정답 확인
② DEFAULT
DEFAULT는 INSERT 시 값이 생략되면 지정한 기본값을 넣는다(CHECK는 값의 허용 범위를 강제). 등록일에 SYSDATE, 상태에 '대기' 같은 기본값을 DB에서 보장하면 SI에서 애플리케이션 누락에도 데이터 일관성이 유지된다.
Q35객관식 · GROUP BY 규칙

SELECT dept_id, name, COUNT(*) FROM EMP GROUP BY dept_id 실행 시 발생하는 문제는?

  1. 정상 실행되어 부서별 이름이 모두 나온다
  2. COUNT(*)가 항상 1이 된다
  3. name이 GROUP BY에도 없고 집계함수도 아니어서 오류(또는 비결정적 결과)가 난다
  4. dept_id가 중복 출력된다
정답 확인
③ name이 GROUP BY에도 없고 집계함수도 아니어서 오류(또는 비결정적 결과)가 난다
GROUP BY 시 SELECT의 컬럼은 그룹 기준이거나 집계함수여야 한다. name은 그룹당 여러 값이라 표준에선 오류다. SI 리포트에서 "부서별 통계에 개별 이름을 같이 넣으려다" 나는 흔한 실수로, 필요하면 별도 조회로 분리한다.
Q36객관식 · ORDER BY 다중정렬

ORDER BY dept_id ASC, salary DESC 의 정렬 방식으로 옳은 것은?

  1. dept_id 오름차순으로 먼저 묶고, 같은 dept_id 안에서 salary 내림차순으로 정렬한다
  2. dept_id와 salary를 모두 오름차순 정렬한다
  3. salary를 먼저 내림차순 정렬한 뒤 dept_id를 정렬한다
  4. 두 컬럼을 합산해 정렬한다
정답 확인
① dept_id 오름차순으로 먼저 묶고, 같은 dept_id 안에서 salary 내림차순으로 정렬한다
다중 정렬은 앞 컬럼이 1순위, 뒤 컬럼이 동점 시 2순위다. "부서별로 묶되 각 부서 안에서 급여 높은 순" 같은 SI 화면 요구의 표준 표현으로, 정렬 컬럼 순서와 각각의 방향을 정확히 지정해야 한다.
Q37쿼리작성 · SELF JOIN

아래 스키마에서 요구사항을 만족하는 SQL을 작성하라.

스키마: EMP(emp_id, name, mgr_id) — mgr_id는 관리자의 emp_id를 가리킨다.

요구사항: 각 사원의 name과 그 사원의 관리자 name(mgr_name)을 함께 조회하라.

모범답안 확인
모범답안
SELECT e.name        AS name,
       m.name        AS mgr_name
FROM   EMP e
JOIN   EMP m ON e.mgr_id = m.emp_id;
같은 EMP 테이블에 e(사원)·m(관리자) 두 별칭을 두고 mgr_id=emp_id로 잇는 셀프 조인이다. 조직 계층 조회의 정석으로, 관리자가 없는 최상위자도 포함하려면 LEFT JOIN으로 바꿔야 한다는 점이 응용 포인트다.
Q38쿼리작성 · 상관 서브쿼리

아래 스키마에서 요구사항을 만족하는 SQL을 작성하라.

스키마: EMP(emp_id, name, dept_id, salary)

요구사항: 각 부서에서 급여가 그 부서의 최고 급여와 같은 사원의 name, dept_id, salary를 조회하라.

모범답안 확인
모범답안
SELECT name, dept_id, salary
FROM   EMP e
WHERE  salary = (SELECT MAX(salary)
                 FROM   EMP
                 WHERE  dept_id = e.dept_id);
서브쿼리가 바깥 행의 dept_id를 참조하는 상관 서브쿼리로, 부서별 기준값을 행마다 계산해 비교한다. "부서별 최고 실적자" 추출은 SI 인사·영업 리포트의 전형이며, 동점자가 여럿이면 모두 나온다는 점을 이해해야 한다.
Q39쿼리작성 · DELETE

아래 스키마에서 요구사항을 만족하는 SQL을 작성하라.

스키마: LOG(log_id, created_at, level)

요구사항: created_at이 2025-01-01 이전인 로그를 삭제하라. (날짜 리터럴은 DATE '2025-01-01' 형식 사용)

모범답안 확인
모범답안
DELETE FROM LOG
WHERE  created_at < DATE '2025-01-01';
DELETE도 WHERE 조건이 없으면 전체 삭제되므로 조건 명시가 필수 채점 포인트다. 오래된 로그·이력 정리는 SI 운영의 일상 배치인데, 실행 전 동일 조건 SELECT로 건수 확인 및 트랜잭션 사용이 사고 예방의 기본이다.
Q40단답 · CROSS JOIN

4행짜리 SIZE 테이블과 3행짜리 COLOR 테이블을 조인 조건 없이 SELECT * FROM SIZE, COLOR (CROSS JOIN)로 조회하면 결과 행수는? 숫자로 답하라.

모범답안 확인
모범답안 · 12
조인 조건이 없으면 두 테이블의 모든 조합(카티션곱)인 4×3=12행이 나온다. SI에서 ON/WHERE 조인 조건을 실수로 빠뜨리면 의도치 않은 카티션곱으로 결과가 폭증하니, 행수가 비정상적으로 크면 이를 먼저 의심한다.

이 세트 우선복습 3줄

  1. GROUP BY 규칙 — SELECT의 비집계 컬럼은 반드시 GROUP BY에 있어야 한다. SELECT dept_id, name, COUNT(*) ... GROUP BY dept_id는 name 때문에 오류.
  2. 상관 서브쿼리 — 서브쿼리가 바깥 행의 컬럼(e.dept_id)을 참조하면 행마다 반복 실행된다. "부서별 최고급여" 같은 행별 기준값 비교의 핵심 패턴.
  3. 다중 정렬 & CROSS JOIN — ORDER BY a ASC, b DESC는 a가 1순위·b가 동점 시 2순위. 조인 조건을 빠뜨리면 카티션곱(4×3=12)으로 행수가 폭증한다.
