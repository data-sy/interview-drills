Q1객관식 · 변수/기본형

자바의 기본형(primitive type)이 아닌 것은?

  1. int
  2. double
  3. boolean
  4. String
정답 확인
④ String
기본형 8종은 byte/short/int/long/float/double/char/boolean이며, String은 참조형이자 불변 객체다. 통계·경영정보 시스템에서 코드값을 String으로 다룰지 int로 다룰지에 따라 비교·형변환 방식이 달라지므로 이 구분이 데이터 처리의 출발점이 된다.
Q2객관식 · 형변환

다음 중 컴파일 에러 없이 자동(묵시적) 형변환이 일어나는 것은?

  1. int i = 3.14;
  2. long l = 10;
  3. int i = 10L;
  4. byte b = 300;
정답 확인
② long l = 10;
작은 타입에서 큰 타입으로(int→long) 갈 때는 손실이 없어 자동 형변환되지만, 반대 방향이나 범위 초과(byte에 300)는 에러 또는 명시적 캐스팅이 필요하다. 고객사 레거시에서 long ID를 int로 받다 오버플로가 나는 장애가 흔해 범위 확인이 습관이 되어야 한다.
Q3객관식 · 연산자(단축평가)

int a=0; if(a!=0 && 10/a>1){...} 에서 10/a는 실행되는가?

  1. 실행되어 ArithmeticException이 발생한다
  2. 컴파일 에러가 난다
  3. 왼쪽이 false라 실행되지 않는다(단축 평가)
  4. 조건이 항상 true가 된다
정답 확인
③ 왼쪽이 false라 실행되지 않는다(단축 평가)
&&는 단축 평가(short-circuit)로 왼쪽이 false면 오른쪽을 평가하지 않아 0으로 나누는 예외가 방지된다. if(obj!=null && obj.isValid()) 같은 null 방어 관용구의 근거이며 SI 코드에서 NPE를 줄이는 기본 패턴이다.
Q4객관식 · 제어문(switch)

break가 없는 switch 문에서 나타나는 현상은?

  1. 컴파일 에러
  2. 매칭된 case 이후 문장이 연달아 실행(fall-through)
  3. default만 실행
  4. 첫 번째 case만 실행
정답 확인
② 매칭된 case 이후 문장이 연달아 실행(fall-through)
break를 생략하면 매칭된 case부터 break를 만나거나 블록이 끝날 때까지 아래 case가 계속 실행된다. 등급·상태 분류 로직에서 의도치 않은 fall-through가 단골 버그라 코드리뷰에서 break 누락을 점검한다.
Q5객관식 · 배열

int[] arr = new int[3]; 직후 arr[0]의 값과 arr.length는?

  1. 0, 3
  2. null, 3
  3. 0, 2
  4. 알 수 없음, 3
정답 확인
① 0, 3
int 배열은 각 요소가 기본값 0으로 초기화되고 length는 요소 개수인 3이다(인덱스 0~2). NPE 없이 0이 들어 있다는 사실은 합계·평균 등 집계 로직을 안전하게 짜는 근거가 된다.
Q6객관식 · 캡슐화

필드를 private로 두고 getter/setter로만 접근하게 하는 주된 목적은?

  1. 실행 속도 향상
  2. 메모리 절약
  3. 내부 구현 은닉과 유효성 검증 지점 확보
  4. 상속 방지
정답 확인
③ 내부 구현 은닉과 유효성 검증 지점 확보
캡슐화는 내부 표현을 숨기고 setter에 검증·로깅 같은 통제점을 둘 수 있게 한다. 고객사 요구로 값 범위 검증이 추가돼도 setter 한 곳만 고치면 되어 유지보수 비용이 줄어든다.
Q7객관식 · 접근제어자

접근 범위가 넓은 것부터 좁은 순서로 옳게 나열한 것은?

  1. protected > public > default > private
  2. public > default > protected > private
  3. private > default > protected > public
  4. public > protected > default > private
정답 확인
④ public > protected > default > private
public(전체) > protected(상속 + 같은 패키지) > default(같은 패키지) > private(같은 클래스) 순이다. 모듈 경계를 넘는 API를 함부로 public으로 열지 말고 좁게 시작하는 것이 대규모 SI 코드의 결합도 관리 핵심이다.
Q8코드읽기 · 정수 나눗셈/형변환

다음 코드의 출력값을 쓰시오.

int a = 7, b = 2;
double r = a / b;
System.out.println(r);
모범답안 확인
모범답안 · 3.0
a / b가 먼저 int 나눗셈으로 3이 되고, 그 결과가 double 변수에 대입되어 3.0이 된다. 3.5를 얻으려면 (double)a / b처럼 나눗셈 전에 캐스팅해야 한다. 정산·비율 계산에서 흔한 버그라 캐스팅 위치를 반드시 확인한다.
Q9단답 · final 지역변수

지역변수 앞에 final을 붙였을 때의 의미를 한 줄로 쓰시오.

모범답안 확인
모범답안
최초 한 번만 값을 할당할 수 있고 이후 재할당(변경)이 불가능한 상수가 된다.

final은 재대입 금지일 뿐 참조형 객체의 내부 필드까지 불변으로 만들지는 않는다. 설정값·상수를 final로 못박아 유지보수 중 실수로 값이 바뀌는 것을 방지한다.
Q10서술형 · OOP 4대 특성

객체지향의 4대 특성을 나열하고 각각을 한 줄로 설명하시오.

모범답안 확인
모범답안
캡슐화(데이터와 기능을 묶고 내부를 은닉), 상속(상위 클래스의 속성·기능을 재사용), 다형성(같은 호출이 실제 객체 타입에 따라 다르게 동작), 추상화(공통 개념만 뽑아 인터페이스/추상클래스로 표현).

채점 포인트는 네 용어의 정확성과 각각의 한 줄 설명이다. 이 특성들이 변경 파급을 국소화해 SI 유지보수에서 재작업 비용을 낮춘다는 점을 덧붙이면 좋다.

이 세트 우선복습 3줄

  1. 단축 평가(&&) — 왼쪽이 false면 오른쪽은 아예 평가되지 않는다. null 방어 관용구 obj!=null && obj.method()의 근거.
  2. 정수 나눗셈 함정 — double r = 7/2;는 3.0. 나눗셈 전에 (double) 캐스팅해야 3.5가 된다.
  3. 접근제어자 범위 — public > protected > default > private. protected는 "상속 + 같은 패키지"라는 점이 헷갈리는 포인트.
