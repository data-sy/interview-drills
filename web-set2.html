Q11객관식 · HTTP 메서드(멱등성)

여러 번 호출해도 결과(서버 상태)가 같은 '멱등한(idempotent)' 메서드가 아닌 것은?

  1. GET
  2. PUT
  3. DELETE
  4. POST
정답 확인
④ POST
GET·PUT·DELETE는 같은 요청을 반복해도 최종 상태가 동일해 멱등하지만, POST는 호출할 때마다 새 리소스가 생겨 멱등하지 않다. SI에서 결제·주문 등록을 POST로 만들 때 사용자 더블클릭에 의한 중복 생성 방지(토큰·유니크키) 설계가 필요한 이유가 이 멱등성 차이다.
Q12객관식 · 상태코드 301 vs 302

301과 302 상태코드의 차이로 옳은 것은?

  1. 301은 임시 이동, 302는 영구 이동이다
  2. 301은 영구 이동, 302는 임시 이동이다
  3. 둘 다 서버 오류를 의미한다
  4. 301은 성공, 302는 인증 실패를 의미한다
정답 확인
② 301은 영구 이동, 302는 임시 이동이다
301(Moved Permanently)은 리소스가 영구 이동해 검색엔진·브라우저가 새 URL을 기억하고, 302(Found)는 임시 이동이라 원래 URL을 유지한다. 고객사 사이트 도메인·URL 개편 시 SEO를 지키려면 301을, 점검 페이지처럼 일시 우회는 302를 써야 한다.
Q13객관식 · 헤더(Content-Type)

HTTP 헤더 Content-Type의 역할로 옳은 것은?

  1. 요청을 보낸 클라이언트의 IP를 나타낸다
  2. 응답 캐시 만료 시간을 지정한다
  3. 본문(body) 데이터의 형식(MIME 타입)을 알려준다
  4. 서버의 인증 토큰을 담는다
정답 확인
③ 본문(body) 데이터의 형식(MIME 타입)을 알려준다
Content-Type은 본문 데이터가 어떤 형식인지(application/json, text/html 등)를 상대에게 알려 파싱 방법을 정하게 한다. API 연동에서 JSON을 보내면서 Content-Type을 잘못 지정하면 서버가 파싱에 실패하므로, SI 인터페이스 오류의 흔한 원인 중 하나다.
Q14객관식 · 쿠키 속성(HttpOnly/Secure)

쿠키 보안 속성에 대한 설명으로 옳은 것은?

  1. HttpOnly는 자바스크립트로 쿠키 접근을 막아 XSS로 인한 탈취를 줄인다
  2. Secure는 HTTP(비암호화)에서만 쿠키를 전송하게 한다
  3. HttpOnly를 켜면 쿠키가 서버로 전송되지 않는다
  4. Secure는 쿠키를 자동으로 암호화 저장한다
정답 확인
① HttpOnly는 자바스크립트로 쿠키 접근을 막아 XSS로 인한 탈취를 줄인다
HttpOnly는 document.cookie 같은 스크립트 접근을 차단해 XSS 공격의 세션 탈취를 줄이고, Secure는 HTTPS에서만 쿠키를 전송한다. 고객사 개인정보·인증 시스템 보안 점검(취약점 진단)에서 세션 쿠키의 HttpOnly·Secure 설정은 단골 체크 항목이다.
Q15객관식 · 서블릿 생명주기

서블릿(Servlet)의 생명주기 메서드 호출 순서로 옳은 것은?

  1. service() → init() → destroy()
  2. init() → destroy() → service()
  3. init() → service() → destroy()
  4. destroy() → init() → service()
정답 확인
③ init() → service() → destroy()
서블릿은 최초 요청 시 한 번 init()으로 초기화되고, 요청마다 service()(→doGet/doPost)가 실행되며, 종료 시 destroy()로 자원을 정리한다. init()이 한 번만 호출된다는 점을 알아야 SI에서 DB 커넥션 등 공유 자원 초기화 위치를 올바르게 잡을 수 있다.
Q16객관식 · Spring DI(@Autowired)

Spring에서 @Autowired 어노테이션의 역할로 옳은 것은?

  1. 클래스를 실행 진입점(main)으로 지정한다
  2. 컨테이너가 관리하는 Bean을 자동으로 주입(DI)한다
  3. 트랜잭션을 강제로 롤백한다
  4. URL과 메서드를 매핑한다
정답 확인
② 컨테이너가 관리하는 Bean을 자동으로 주입(DI)한다
@Autowired는 스프링 컨테이너가 관리하는 Bean을 타입 기준으로 찾아 자동 주입(의존성 주입)한다. 객체 생성을 직접 하지 않으므로 결합도가 낮아지고 테스트·교체가 쉬워, SI에서 여러 모듈을 조립·유지보수하는 표준 방식이 된다.
Q17객관식 · AJAX(비동기)

AJAX에 대한 설명으로 옳은 것은?

  1. 페이지 전체를 다시 로드하지 않고 필요한 데이터만 비동기로 주고받아 화면 일부를 갱신한다
  2. 페이지 전체를 새로고침해야만 서버 데이터를 받는다
  3. 서버 없이 브라우저 안에서만 동작한다
  4. HTML을 CSS로 변환하는 기술이다
정답 확인
① 페이지 전체를 다시 로드하지 않고 필요한 데이터만 비동기로 주고받아 화면 일부를 갱신한다
AJAX는 자바스크립트로 서버와 비동기 통신해 전체 페이지 리로드 없이 부분 갱신을 한다. 검색 자동완성, 목록 무한스크롤처럼 사용자 체감 속도를 높이며, SI 통계·조회 화면에서 대량 데이터를 페이지 전환 없이 갱신할 때 널리 쓰인다.
Q18서술형 · 쿠키 vs 세션

쿠키(Cookie)와 세션(Session)의 차이를 저장 위치·보안·수명 관점에서 비교 설명하시오.

모범답안 확인
모범답안
쿠키는 클라이언트(브라우저)에 저장돼 매 요청에 실려 가므로 탈취·위변조 위험이 있고, 세션은 데이터를 서버에 저장하고 세션ID만 쿠키로 오가 상대적으로 안전하다. 수명 면에서 쿠키는 만료시간을 지정해 브라우저 종료 후에도 유지할 수 있고, 세션은 보통 브라우저 종료나 타임아웃 시 소멸한다. 서버 자원 소모는 세션이 더 크다.

채점 포인트는 저장 위치(클라이언트/서버), 보안 차이, 수명, 서버 부하다. SI에서 로그인 상태는 세션으로, '자동 로그인·언어 설정' 같은 비민감 정보는 쿠키로 두는 식의 역할 분담을 설계할 수 있어야 한다.
Q19서술형 · MVC(도입 이유)

웹 개발에서 MVC 패턴을 사용하는 이유(장점)를 설명하시오.

모범답안 확인
모범답안
MVC는 데이터·로직(Model), 화면(View), 흐름 제어(Controller)로 책임을 분리하는 패턴이다. 관심사가 분리돼 화면을 바꿔도 로직에 영향이 적고, 각 부분을 독립적으로 개발·테스트·재사용할 수 있어 유지보수성과 협업 효율이 높아진다. 또 화면 담당(퍼블리셔)과 로직 담당(개발자)의 분업이 명확해진다.

채점 포인트는 관심사 분리, 유지보수·재사용성, 분업 용이성이다. 요구·화면 변경이 잦은 SI 특성상 계층 분리가 안 되면 작은 UI 수정이 로직 전반에 파급되므로, MVC(및 Spring MVC)가 사실상 표준 구조로 쓰인다.
Q20서술형 · CORS

CORS(Cross-Origin Resource Sharing)가 무엇이며 왜 발생하는지 설명하시오.

모범답안 확인
모범답안
브라우저는 보안을 위해 동일 출처 정책(Same-Origin Policy)으로 다른 출처(프로토콜·도메인·포트가 다름)의 리소스 요청을 기본 차단한다. CORS는 서버가 응답 헤더(Access-Control-Allow-Origin 등)로 특정 출처의 접근을 명시적으로 허용해주는 표준 방식이다. 즉 프론트엔드와 API 서버의 출처가 다를 때 서버가 허용 헤더를 내려줘야 브라우저가 응답을 사용할 수 있다.

채점 포인트는 동일 출처 정책, 출처(origin)의 정의, 서버가 허용 헤더로 푸는 구조다. SI에서 프론트와 API를 분리 배포하거나 외부사 API를 붙일 때 CORS 오류가 자주 나므로, 서버 측 허용 설정으로 해결한다는 점을 알아야 한다.

이 세트 우선복습 3줄

  1. 멱등성 — GET·PUT·DELETE는 반복해도 상태가 같지만 POST는 멱등하지 않다. 결제·등록의 중복 생성 방지 설계가 필요한 근거.
  2. 301 vs 302 — 301은 영구 이동(SEO·새 URL 기억), 302는 임시 이동(원래 URL 유지). 방향이 헷갈리는 단골 함정.
  3. 서블릿 생명주기 — init()(1회) → service()(매 요청) → destroy()(종료). init이 한 번뿐이라는 점이 공유 자원 초기화 위치의 기준.
