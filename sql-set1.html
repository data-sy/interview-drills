SELECT·정렬·집계부터 조인·NULL까지

보기를 고른 뒤 채점하기를 누르면 정오답과 해설이 열립니다. 쿼리작성·단답은 직접 적어보고 모범답안과 비교해 스스로 채점하세요.

Q1객관식 · SELECT/ORDER BY

ORDER BY 절에서 정렬 방향을 생략(ORDER BY salary)했을 때 기본 정렬 방향은?

  1. 내림차순(DESC)
  2. 입력된 순서 그대로
  3. 오름차순(ASC)
  4. DBMS마다 랜덤
정답 확인
③ 오름차순(ASC)
방향을 안 적으면 표준상 오름차순(ASC)이 기본이다. SI 통계·리포트 화면에서 "매출 높은 순"을 원하는데 DESC를 빼먹어 낮은 순으로 나오는 실수가 잦으니, 정렬 컬럼마다 방향을 명시하는 습관이 유지보수에 안전하다.
Q2객관식 · DISTINCT

SELECT DISTINCT dept_id, job FROM emp 의 동작으로 옳은 것은?

  1. dept_id와 job의 조합이 동일한 행을 하나로 합친다
  2. dept_id 컬럼만 중복 제거하고 job은 그대로 둔다
  3. 첫 번째 컬럼 기준으로만 정렬한다
  4. NULL은 서로 다른 값으로 취급해 제거되지 않는다
정답 확인
① dept_id와 job의 조합이 동일한 행을 하나로 합친다
DISTINCT는 뒤에 나열된 컬럼 조합 전체를 기준으로 중복을 제거한다. 고객사 코드성 데이터에서 "실제로 존재하는 부서-직무 조합"을 뽑을 때 자주 쓰며, 한 컬럼만 유일하게 만드는 것으로 오해하면 중복 통계가 나온다.
Q3객관식 · 집계함수·NULL

comm 컬럼에 NULL이 3건 섞인 10행 테이블에서 COUNT(*)와 COUNT(comm)의 결과는?

  1. 둘 다 10
  2. COUNT(*)=7, COUNT(comm)=10
  3. 둘 다 7
  4. COUNT(*)=10, COUNT(comm)=7
정답 확인
④ COUNT(*)=10, COUNT(comm)=7
COUNT(*)는 행 수를 세지만 COUNT(컬럼)은 그 컬럼이 NULL인 행을 제외한다. 성과급(comm)처럼 미입력이 많은 컬럼을 집계할 때 이 차이를 모르면 "지급 대상 인원" 통계가 틀어지므로 SI 정산 리포트에서 특히 주의해야 한다.
Q4객관식 · WHERE vs HAVING

부서별 인원이 5명을 초과하는 부서만 조회하려 한다. "인원수 > 5" 조건은 어디에 써야 하는가?

  1. WHERE COUNT(*) > 5
  2. GROUP BY 뒤 HAVING COUNT(*) > 5
  3. ORDER BY COUNT(*) > 5
  4. SELECT 절 안
정답 확인
② GROUP BY 뒤 HAVING COUNT(*) > 5
WHERE는 그룹핑 전 개별 행을, HAVING은 그룹핑 후 집계 결과를 거른다. 집계함수 조건은 반드시 HAVING이다. SI에서 "특정 규모 이상 부서/거래처만" 뽑는 요구가 흔한데, WHERE에 COUNT를 쓰면 문법 오류가 난다.
Q5객관식 · LIKE

이름이 '김'으로 시작하는 사원을 찾는 조건으로 올바른 것은?

  1. WHERE name = '김%'
  2. WHERE name IN '김'
  3. WHERE name LIKE '김%'
  4. WHERE name LIKE '%김'
정답 확인
③ WHERE name LIKE '김%'
%는 0글자 이상 임의 문자열을 뜻하며 접두 검색은 '김%'다. =는 와일드카드를 문자 그대로 비교한다. 고객사 회원/거래처 검색 화면의 "성으로 찾기"에 쓰이며, 앞에 %를 붙이면 인덱스를 못 타 성능이 급락한다는 점도 함께 기억한다.
Q6객관식 · DDL/DML/DCL/TCL

다음 중 DDL(데이터 정의어)에 속하며, 롤백이 불가능한(자동 커밋) 명령은?

  1. TRUNCATE
  2. DELETE
  3. UPDATE
  4. SELECT
정답 확인
① TRUNCATE
TRUNCATE는 테이블 구조는 두고 전체 행을 제거하는 DDL로, 대부분의 DBMS에서 자동 커밋되어 롤백이 안 된다. DML인 DELETE는 트랜잭션으로 되돌릴 수 있다. 운영 DB에서 대량 초기화 시 이 차이를 모르면 복구 불가능한 사고로 이어진다.
Q7쿼리작성 · SELECT/WHERE/ORDER BY

아래 스키마에서 요구사항을 만족하는 SQL을 작성하라.
스키마: EMP(emp_id, name, dept_id, salary, hire_date)
요구사항: dept_id가 10인 사원 중 salary가 3000 이상인 사람의 name과 salary를, salary 높은 순으로 조회하라.

모범답안 확인
모범답안
SELECT name, salary
FROM   EMP
WHERE  dept_id = 10
  AND  salary >= 3000
ORDER  BY salary DESC;
WHERE에 AND로 두 조건을 결합하고 정렬은 DESC로 명시하는 것이 채점 포인트다. SI 인사/급여 조회 화면의 가장 기본 골격으로, 조건 누락이나 정렬 방향 실수가 실제 화면 버그로 직결된다.
Q8쿼리작성 · GROUP BY

아래 스키마에서 요구사항을 만족하는 SQL을 작성하라.
스키마: EMP(emp_id, name, dept_id, salary)
요구사항: 부서(dept_id)별 사원 수와 평균 급여를 구하되, 평균 급여는 avg_sal 별칭으로 출력하라.

모범답안 확인
모범답안
SELECT dept_id,
       COUNT(*)      AS emp_cnt,
       AVG(salary)   AS avg_sal
FROM   EMP
GROUP  BY dept_id;
집계함수와 함께 SELECT에 나오는 비집계 컬럼(dept_id)은 반드시 GROUP BY에 포함해야 한다. 부서·거래처·월별 집계는 SI 경영정보 리포트의 핵심이며, 별칭(AS)은 화면 헤더 매핑에 그대로 쓰인다.
Q9쿼리작성 · INNER JOIN

아래 스키마에서 요구사항을 만족하는 SQL을 작성하라.
스키마: EMP(emp_id, name, dept_id), DEPT(dept_id, dept_name)
요구사항: 각 사원의 name과 소속 부서명(dept_name)을 함께 조회하라. 부서가 없는 사원은 제외한다.

모범답안 확인
모범답안
SELECT e.name, d.dept_name
FROM   EMP e
JOIN   DEPT d ON e.dept_id = d.dept_id;
조인 조건(ON)에 두 테이블의 공통 키를 걸고, "부서 없는 사원 제외"는 INNER JOIN의 자연스러운 동작이다. 코드-명칭 매핑(코드성 테이블 조인)은 SI에서 가장 흔한 조인 패턴으로, ON 누락 시 카티션곱이 발생한다.
Q10단답 · NULL 산술

comm이 NULL인 사원에 대해 SELECT salary + comm FROM EMP 를 실행하면 그 사원의 결과값은 무엇인가? 한 단어로 답하라.

모범답안 확인
모범답안 · NULL
NULL은 "값 없음/미지수"라서 어떤 값과 산술연산해도 결과가 NULL이 된다. SI 급여 합산에서 이를 방치하면 총지급액이 통째로 NULL로 비어 정산이 깨지므로 salary + NVL(comm,0)처럼 NULL을 0으로 치환해야 한다.

이 세트 우선복습 3줄

  1. COUNT(*) vs COUNT(컬럼) — COUNT(*)는 행 수, COUNT(comm)은 NULL을 뺀 개수. 미입력 많은 컬럼 집계에서 인원수가 틀어지는 함정.
  2. WHERE vs HAVING — 집계함수 조건(COUNT(*) > 5)은 WHERE가 아니라 GROUP BY 뒤 HAVING에. WHERE에 쓰면 문법 오류.
  3. NULL 산술 = NULL — salary + comm에서 comm이 NULL이면 전체가 NULL. NVL(comm,0)로 치환해야 합계가 깨지지 않는다.
