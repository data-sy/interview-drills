Q31객관식 · async/await Task

비동기 프로그래밍 모델의 자바↔C# 대비로 옳은 것은?

  1. C#에는 비동기 개념이 없어 항상 스레드를 직접 만들어야 한다
  2. 자바 Future/CompletableFuture에 대응해 C#은 Task/Task<T>와 async/await 문법으로 비동기를 표현한다
  3. async/await는 자바 키워드이고 C#은 CompletableFuture만 쓴다
  4. C# Task는 항상 새 스레드를 만들어 병렬 실행한다
정답 확인
② 자바 Future/CompletableFuture에 대응해 C#은 Task/Task<T>와 async/await 문법으로 비동기를 표현한다
자바의 Future/CompletableFuture 비동기 결과에 대응해, C#은 Task/Task<T>를 async/await 문법으로 다뤄 비동기 코드를 동기처럼 읽히게 쓴다. await가 반드시 새 스레드를 만드는 건 아니다(I/O 대기 반납). 외부 API·DB 호출이 많은 웹 서비스에서 스레드를 놀리지 않아 처리량이 올라간다.
Q32객관식 · .NET Framework vs .NET

.NET Framework와 .NET(Core 이후) 및 자바 대비로 옳은 것은?

  1. 자바가 JVM/JDK로 크로스플랫폼이듯, .NET(Core 이후)은 크로스플랫폼이고 구형 .NET Framework는 주로 Windows 전용이다
  2. .NET Framework가 최신 크로스플랫폼이고 .NET Core가 구형 Windows 전용이다
  3. .NET은 자바와 달리 런타임(가상머신)이 전혀 없이 기계어로만 실행된다
  4. 자바의 JVM에 해당하는 것이 .NET에는 없다
정답 확인
① 자바가 JVM/JDK로 크로스플랫폼이듯, .NET(Core 이후)은 크로스플랫폼이고 구형 .NET Framework는 주로 Windows 전용이다
자바가 JVM 위에서 크로스플랫폼으로 돌듯, 신규 .NET(Core→.NET 5/6+)은 Windows/Linux/macOS 크로스플랫폼이고 구형 .NET Framework는 Windows 전용이다. .NET에도 CLR이라는 런타임이 있어 JVM 역할을 한다. 신규 SI 프로젝트는 리눅스 배포·컨테이너를 위해 최신 .NET을 택하는 추세다.
Q33객관식 · get-only/init 프로퍼티

불변 객체를 만드는 프로퍼티 기능의 자바↔C# 대비로 옳은 것은?

  1. C#은 불변 객체를 못 만들어 모든 프로퍼티가 항상 set 가능하다
  2. init 접근자는 객체 생성 후에도 언제든 값을 바꿀 수 있다
  3. 자바 final과 달리 C#엔 불변을 위한 어떤 수단도 없다
  4. 자바는 final 필드+생성자로 불변을 만들고, C#은 get-only 자동 프로퍼티나 init 접근자(초기화 시에만 설정)로 불변을 표현한다
정답 확인
④ 자바는 final 필드+생성자로 불변을 만들고, C#은 get-only 자동 프로퍼티나 init 접근자(초기화 시에만 설정)로 불변을 표현한다
자바는 final 필드 + 생성자 주입으로 불변을 만들고, C#은 { get; }(get-only)나 { get; init; }(객체 초기화 시점에만 설정)로 같은 목적을 달성한다. init은 생성 이후 변경을 막는다. 설정값·DTO 등 생성 후 바뀌면 안 되는 객체에 불변을 걸어 예측 가능성과 스레드 안전성을 높인다.
Q34객관식 · out/ref 매개변수

메서드 인자 전달의 자바↔C# 차이로 옳은 것은?

  1. 자바도 ref 키워드로 변수를 참조 전달할 수 있다
  2. C#의 out은 자바 return과 완전히 동일해 아무 차이가 없다
  3. 자바는 항상 값 전달(참조 타입은 참조값을 복사)만 되지만, C#은 ref/out 키워드로 변수 자체를 참조로 넘겨 호출부 변수를 바꿀 수 있다
  4. C#은 참조 전달만 되고 값 전달이 불가능하다
정답 확인
③ 자바는 항상 값 전달(참조 타입은 참조값을 복사)만 되지만, C#은 ref/out 키워드로 변수 자체를 참조로 넘겨 호출부 변수를 바꿀 수 있다
자바는 값 전달만 지원해 원시값이든 참조값이든 복사본이 넘어간다. C#은 ref(초기화된 변수를 참조로)와 out(메서드 내에서 반드시 대입, 여러 값 반환용)으로 호출부 변수 자체를 바꿀 수 있다. TryParse처럼 성공 여부+결과를 함께 돌려주는 실무 API가 out을 쓴다.
Q35객관식 · enum

열거형(enum)의 자바↔C# 대비로 옳은 것은?

  1. C# enum은 자바처럼 각 상수마다 메서드 본문을 정의하는 완전한 클래스다
  2. 자바 enum은 메서드·필드를 가질 수 있는 완전한 클래스지만, C# enum은 기본적으로 정수형(int 등) 기반의 이름 붙은 상수 집합이다
  3. 자바 enum과 C# enum 모두 문자열 리터럴 기반이라 정수 변환이 불가능하다
  4. enum은 자바에만 있고 C#은 const 묶음으로만 대체한다
정답 확인
② 자바 enum은 메서드·필드를 가질 수 있는 완전한 클래스지만, C# enum은 기본적으로 정수형(int 등) 기반의 이름 붙은 상수 집합이다
자바 enum은 상수마다 동작·필드를 가질 수 있는 강력한 클래스형이지만, C# enum은 기본적으로 int 등 정수형에 이름을 붙인 경량 상수 집합이라 정수 캐스팅이 자유롭다(C#은 동작이 필요하면 확장 메서드 등으로 보완). 상태 코드·구분값 매핑에 쓰되, C# enum은 정의 안 된 정수도 담길 수 있어 검증이 필요하다.
Q36객관식 · 문자열 보간

변수를 문자열에 끼워 넣는 방식의 자바↔C# 대비로 옳은 것은?

  1. 자바는 String.format("%s", x)나 + 연결을, C#은 문자열 보간 $"{x}" 문법을 제공한다
  2. C#에는 문자열 포매팅 기능이 없어 항상 + 로만 연결한다
  3. 자바 String.format과 C# $"..."는 문법이 완전히 동일하다
  4. 문자열 보간은 자바 전용이고 C#은 지원하지 않는다
정답 확인
① 자바는 String.format("%s", x)나 + 연결을, C#은 문자열 보간 $"{x}" 문법을 제공한다
자바는 String.format("%s살", age)나 + 연결로 값을 끼우고, C#은 $"{age}살" 보간 문법으로 변수를 인라인한다(C#에도 string.Format 존재). 보간은 가독성이 좋고 오타를 줄인다. 로그 메시지·화면 문구 조립에서 자주 쓰여 유지보수 가독성을 높인다.
Q37객관식 · 확장 메서드

기존 타입에 메서드를 덧붙이는 C# 확장 메서드에 대한 자바 대비 설명으로 옳은 것은?

  1. 확장 메서드는 기존 타입의 소스를 직접 수정해 메서드를 넣는 기능이다
  2. 자바에도 확장 메서드 키워드가 있어 C#과 문법이 같다
  3. C#은 유틸리티 정적 메서드를 아예 만들 수 없다
  4. 자바는 별도 유틸리티 클래스의 정적 메서드로 기능을 보완하지만, C#은 확장 메서드로 기존 타입에 인스턴스 메서드처럼 보이는 기능을 추가한다
정답 확인
④ 자바는 별도 유틸리티 클래스의 정적 메서드로 기능을 보완하지만, C#은 확장 메서드로 기존 타입에 인스턴스 메서드처럼 보이는 기능을 추가한다
자바는 StringUtils.isEmpty(s)처럼 유틸 클래스 정적 메서드로 보완하지만, C#은 확장 메서드로 s.IsEmpty()처럼 원래 타입에 메서드가 있는 듯 호출하게 한다(실제로는 정적 메서드, this 매개변수). 원본 타입을 못 고치는 외부 라이브러리 타입에 팀 공용 헬퍼를 붙일 때 유용하다.
Q38객관식 · protected internal

C#의 protected internal 접근제어자에 대한 자바 대비 설명으로 옳은 것은?

  1. 같은 어셈블리이면서 "동시에" 파생 클래스일 때만 접근 가능하다(교집합)
  2. 자바 protected와 완전히 동일해 같은 패키지에서만 접근된다
  3. 자바엔 없는 조합으로, 같은 어셈블리(internal) "또는" 파생 클래스(protected) 어디서든 접근 가능하다(합집합)
  4. private보다 좁은 최소 접근 범위를 뜻한다
정답 확인
③ 자바엔 없는 조합으로, 같은 어셈블리(internal) "또는" 파생 클래스(protected) 어디서든 접근 가능하다(합집합)
C#의 protected internal은 protected(파생 클래스) "또는" internal(같은 어셈블리)의 합집합 접근이다(교집합은 별도 키워드 private protected). 자바엔 이런 조합 자체가 없고 protected가 같은 패키지+하위 클래스를 커버하는 등 의미가 다르다. 라이브러리 확장점을 설계할 때 노출 범위를 정확히 지정하는 데 쓴다.
Q39코드읽기 · async 실행 순서

다음 C# 코드의 출력 순서는? 자바 CompletableFuture와 대비하며 읽어라.

async Task RunAsync() {
    Console.WriteLine("A");
    await Task.Delay(100);
    Console.WriteLine("C");
}
Console.WriteLine("start");
var t = RunAsync();
Console.WriteLine("B");
await t;
모범답안 확인
모범답안 · start → A → B → C
RunAsync 호출 시 await 전까지 동기 실행되어 "A"가 먼저 찍히고, await Task.Delay에서 제어가 호출부로 반환돼 "B"가 찍힌다. 지연이 끝나면 "C"가 이어진다(await t가 완료를 기다림). 자바 CompletableFuture 체인과 개념은 같으나 async/await는 흐름이 순차 코드처럼 읽힌다. 외부 호출 대기 중 다른 일을 처리하는 웹 서비스의 핵심 동작이다.
Q40단답 · object 최상위

자바의 모든 클래스의 최상위 부모는 java.lang.Object다. C#에서 이에 대응하는 모든 타입의 최상위 기반 타입을 키워드와 정식 이름으로 쓰시오.

모범답안 확인
모범답안 · object (정식 이름 System.Object)
자바의 java.lang.Object에 대응하는 C#의 최상위 타입은 object(별칭)이며 정식 이름은 System.Object다. ToString/Equals/GetHashCode 등 공통 메서드도 대응된다(자바 toString ↔ C# ToString). 공용 로깅·직렬화 유틸이 object 파라미터로 임의 타입을 받는 근거가 된다.

이 세트 우선복습 3줄

  1. async/await ≠ 새 스레드 — await는 I/O 대기 중 제어를 반납할 뿐 반드시 스레드를 새로 만들지 않는다. 출력 순서는 start→A→B→C.
  2. ref/out vs 자바 값 전달 — 자바는 언제나 값(참조값 복사) 전달, C#은 ref/out으로 호출부 변수 자체를 바꾼다. TryParse의 out이 대표 예.
  3. protected internal은 합집합 — internal 또는 protected(어느 한쪽만 만족해도 접근). 교집합은 별도 키워드 private protected.
