Q11객관식 · LEFT JOIN

EMP LEFT JOIN DEPT ON e.dept_id = d.dept_id 의 결과에 대한 설명으로 옳은 것은?

  1. 양쪽 모두에 매칭되는 행만 나온다
  2. 부서 테이블의 모든 행이 기준이 된다
  3. 매칭 안 되면 그 사원은 제외된다
  4. 부서가 없는(dept_id가 NULL이거나 매칭 안 되는) 사원도 나오고, 그 부서 컬럼은 NULL로 채워진다
정답 확인
④ 부서가 없는 사원도 나오고, 그 부서 컬럼은 NULL로 채워진다
LEFT JOIN은 왼쪽(EMP) 전체를 보존하고 오른쪽에 짝이 없으면 NULL로 채운다. "주문이 없는 고객까지 포함한 명단" 같이 누락을 살려야 하는 SI 요구에 필수이며, INNER JOIN으로 짜면 대상이 조용히 빠진다.
Q12객관식 · HAVING

다음 쿼리의 결과로 옳은 것은? SELECT dept_id, AVG(salary) FROM EMP GROUP BY dept_id HAVING AVG(salary) >= 4000

  1. 모든 부서의 평균 급여를 보여준다
  2. 평균 급여가 4000 이상인 부서만 보여준다
  3. 급여가 4000 이상인 개별 사원을 보여준다
  4. 문법 오류가 난다
정답 확인
② 평균 급여가 4000 이상인 부서만 보여준다
HAVING은 GROUP BY로 만든 그룹에 집계 조건을 걸어 그룹 단위로 걸러낸다. "평균 실적이 기준 이상인 팀만" 같은 SI 성과 리포트에 직결되며, 같은 조건을 WHERE에 쓰면 개별 사원 필터가 되어 의미가 달라진다.
Q13객관식 · 제약조건 PK

기본키(PRIMARY KEY) 제약에 대한 설명으로 옳은 것은?

  1. NULL을 허용하지만 중복은 불가하다
  2. 중복은 허용하지만 NULL은 불가하다
  3. NULL과 중복 둘 다 허용한다
  4. NULL 불가 + 유일성(중복 불가)을 동시에 보장한다
정답 확인
④ NULL 불가 + 유일성(중복 불가)을 동시에 보장한다
PK는 UNIQUE와 NOT NULL을 합친 것으로 행을 유일하게 식별한다. SI에서 회원ID·주문번호 등 식별자에 걸며, PK가 없으면 중복 데이터·부정확한 조인으로 유지보수 시 원인 추적이 매우 어려워진다.
Q14객관식 · NVL/COALESCE

NVL(comm, 0) (오라클)의 동작으로 옳은 것은?

  1. comm이 NULL이면 0으로 바꾸고, 아니면 comm 값을 그대로 쓴다
  2. comm이 0이면 NULL로 바꾼다
  3. comm을 무조건 0으로 만든다
  4. comm이 NULL이면 오류를 낸다
정답 확인
① comm이 NULL이면 0으로 바꾸고, 아니면 comm 값을 그대로 쓴다
NVL(값, 대체값)은 첫 인자가 NULL일 때만 대체값을 반환한다(표준 SQL은 COALESCE). SI 통계에서 미입력 항목을 0이나 '미정'으로 채워 합계·화면 표시를 안정화하는 데 필수적인 함수다.
Q15객관식 · UNION vs UNION ALL

UNION과 UNION ALL의 차이로 옳은 것은?

  1. UNION은 중복을 제거하고, UNION ALL은 중복을 그대로 둔다
  2. UNION ALL이 항상 중복을 제거한다
  3. 둘 다 중복을 제거하지만 정렬만 다르다
  4. UNION은 컬럼 개수가 달라도 된다
정답 확인
① UNION은 중복을 제거하고, UNION ALL은 중복을 그대로 둔다
UNION은 정렬·중복제거 과정이 있어 느리고, UNION ALL은 그냥 이어붙여 빠르다. 중복이 없음을 아는 데이터(예: 월별로 나뉜 로그 통합)라면 SI에서 성능을 위해 UNION ALL을 선택하는 것이 정석이다.
Q16객관식 · CASE WHEN

다음 표현의 결과로 옳은 것은? CASE WHEN salary >= 5000 THEN 'A' WHEN salary >= 3000 THEN 'B' ELSE 'C' END (salary가 3500일 때)

  1. 'A'
  2. 'C'
  3. 'B'
  4. NULL
정답 확인
③ 'B'
CASE는 위에서부터 첫 번째로 참인 WHEN을 골라 멈춘다. 3500은 5000 미만이라 첫 조건은 거짓, 3000 이상이라 'B'다. SI 리포트에서 점수·금액을 등급/구간으로 변환할 때 조건 순서를 큰 값부터 배치하는 것이 핵심이다.
Q17쿼리작성 · LEFT JOIN + NULL

아래 스키마에서 요구사항을 만족하는 SQL을 작성하라.
스키마: CUSTOMER(cust_id, name), ORDERS(order_id, cust_id, amount)
요구사항: 아직 주문 이력이 한 건도 없는 고객의 name을 조회하라.

모범답안 확인
모범답안
SELECT c.name
FROM   CUSTOMER c
LEFT   JOIN ORDERS o ON c.cust_id = o.cust_id
WHERE  o.order_id IS NULL;
LEFT JOIN으로 고객 전체를 살린 뒤 오른쪽이 NULL인 행만 남기면 "매칭 실패=주문 없음" 고객이 걸러진다. 휴면·미거래 대상 추출은 SI 마케팅/정산 요건에 자주 나오는 안티조인 패턴이다.
Q18쿼리작성 · HAVING

아래 스키마에서 요구사항을 만족하는 SQL을 작성하라.
스키마: ORDERS(order_id, cust_id, amount)
요구사항: 고객(cust_id)별 총 주문금액이 100000 이상인 고객의 cust_id와 총액(total_amt)을 조회하라.

모범답안 확인
모범답안
SELECT cust_id, SUM(amount) AS total_amt
FROM   ORDERS
GROUP  BY cust_id
HAVING SUM(amount) >= 100000;
그룹별 합계 조건은 WHERE가 아니라 HAVING에 SUM으로 건다. "우수 거래처 선별" 같은 SI 요구의 전형으로, GROUP BY와 HAVING의 역할 분리를 정확히 이해했는지가 채점 포인트다.
Q19쿼리작성 · UPDATE

아래 스키마에서 요구사항을 만족하는 SQL을 작성하라.
스키마: EMP(emp_id, name, dept_id, salary)
요구사항: dept_id가 20인 사원의 salary를 10% 인상하라.

모범답안 확인
모범답안
UPDATE EMP
SET    salary = salary * 1.1
WHERE  dept_id = 20;
UPDATE에서 WHERE를 빠뜨리면 전체 행이 갱신되는 대형 사고가 난다. SI 운영 반영 작업 전 항상 같은 조건의 SELECT로 대상 건수를 먼저 확인하고, 트랜잭션 안에서 실행 후 커밋하는 것이 안전 관행이다.
Q20단답 · UNION ALL

A 테이블에 4행, B 테이블에 3행이 있고 두 결과에 중복이 2건 있다. SELECT ... FROM A UNION ALL SELECT ... FROM B 의 결과 행수는 몇 행인가? 숫자로 답하라.

모범답안 확인
모범답안 · 7
UNION ALL은 중복을 제거하지 않고 그대로 이어붙이므로 4+3=7행이다(같은 조건에서 UNION이면 중복 2건이 합쳐져 5행). 로그·이력 통합 시 건수가 맞지 않으면 이 중복제거 여부를 먼저 의심한다.

이 세트 우선복습 3줄

  1. LEFT JOIN + IS NULL = 안티조인 — 왼쪽 전체를 살린 뒤 o.order_id IS NULL로 "매칭 실패=주문 없음" 고객을 뽑는다. INNER JOIN으로 짜면 대상이 조용히 빠진다.
  2. 집계 조건은 WHERE가 아니라 HAVING — HAVING SUM(amount) >= 100000처럼 GROUP BY 후 그룹 단위로 거른다. 같은 조건을 WHERE에 쓰면 개별 행 필터가 되어 의미가 달라진다.
  3. UNION vs UNION ALL — UNION은 중복 제거(정렬 비용) , UNION ALL은 그대로 이어붙여 빠르다. 4행+3행 = UNION ALL은 7행, UNION은 중복 2건 합쳐 5행.
