Q11객관식 · struct vs class

자바와 C#의 값 형식/참조 형식 대비로 옳은 것은?

  1. C#의 class는 값 형식이고 struct는 참조 형식이다
  2. 자바에도 struct 키워드가 있어 사용자 정의 값 형식을 만든다
  3. C#의 int는 참조 형식이라 항상 힙에 저장된다
  4. 자바는 int 등 원시타입만 값 형식이고 나머지는 참조 형식이지만, C#은 struct로 사용자 정의 값 형식을 만들 수 있다
정답 확인
④ 자바는 원시타입만 값 형식, C#은 struct로 사용자 정의 값 형식
자바는 원시타입(int, double 등)만 값 형식이고 사용자 정의 타입은 모두 참조 형식이다. C#은 struct로 사용자 정의 값 형식(대입 시 복사, 주로 스택/인라인 저장)을 만들 수 있고 class는 참조 형식이다. 좌표·금액 같은 작고 불변인 값에 struct를 쓰면 힙 할당과 GC를 줄일 수 있다.
Q12객관식 · virtual/override

메서드 재정의(오버라이딩) 시 자바와 C#의 차이로 옳은 것은?

  1. 자바도 virtual 키워드를 붙여야만 재정의된다
  2. 자바는 메서드가 기본으로 재정의 가능하지만, C#은 기반 메서드에 virtual, 파생에 override를 명시해야 재정의된다
  3. C#은 아무 키워드 없이도 자바처럼 자동으로 재정의된다
  4. C#의 override는 기반 메서드가 sealed일 때만 동작한다
정답 확인
② 자바는 기본 재정의 가능, C#은 virtual+override 명시 필요
자바 인스턴스 메서드는 기본이 virtual이라 그냥 재정의되지만(final이면 금지), C#은 기반에 virtual, 파생에 override를 명시해야 다형성이 성립한다. 키워드를 빼면 숨김(new)이 돼 버그가 된다. 프레임워크 확장 지점을 설계할 때 virtual 지정 여부가 확장 가능 API의 경계가 된다.
Q13객관식 · sealed

자바의 final class에 대응하는 C# 키워드는?

  1. sealed — 더 이상 상속할 수 없는 클래스
  2. static — 인스턴스화할 수 없는 클래스
  3. abstract — 반드시 상속해야 하는 클래스
  4. const — 상수 클래스
정답 확인
① sealed — 더 이상 상속할 수 없는 클래스
자바 final class(상속 금지)에 대응하는 C# 키워드는 sealed class다. 메서드 단위로도 자바 final method ↔ C# sealed override로 확장을 봉인한다. 보안·정합성이 중요한 결제/정산 클래스를 sealed로 잠가 고객사 커스터마이징 중 예기치 않은 상속 변경을 차단한다.
Q14객관식 · nullable value type

값 타입에 null을 담는 방법의 자바↔C# 대비로 옳은 것은?

  1. 자바 int와 C# int 모두 그냥 null을 대입할 수 있다
  2. C#은 값 타입에 null을 못 담아 항상 object로 박싱해야 한다
  3. 자바는 int 대신 래퍼 Integer로 null을 표현하고, C#은 int?(Nullable<int>)로 값 타입에 null을 허용한다
  4. 자바 Integer와 달리 C# int?는 참조 형식이라 힙에 저장된다
정답 확인
③ 자바는 래퍼 Integer, C#은 int?(Nullable<int>)
자바는 원시 int에 null을 못 넣어 래퍼 Integer를 쓰고, C#은 int?(=Nullable<int>) 값 형식으로 null 가능 여부를 표현한다. int?는 여전히 값 형식이다. DB의 nullable 숫자 컬럼을 매핑할 때 C#의 int?가 자바 Integer 역할을 하며, HasValue 확인을 빠뜨리면 런타임 예외가 난다.
Q15객관식 · Dictionary vs HashMap

자바 HashMap<K,V>에 대응하는 C# 표준 맵 타입은?

  1. Dictionary<TKey, TValue>
  2. HashMap<TKey, TValue>
  3. Map<TKey, TValue>
  4. Hashtable<TKey, TValue>
정답 확인
① Dictionary<TKey, TValue>
자바 HashMap<K,V>에 대응하는 C#의 표준 제네릭 맵은 Dictionary<TKey,TValue>다. 자바는 없는 키 조회 시 null을 주지만 C# Dictionary는 인덱서 접근 시 KeyNotFoundException을 던지므로 TryGetValue/ContainsKey로 확인한다. 코드 조회·매핑 로직에서 이 예외 차이를 모르면 이관 후 장애가 난다.
Q16객관식 · 예외 checked 여부

예외 처리 강제 여부의 자바↔C# 차이로 옳은 것은?

  1. C#도 자바처럼 throws 절로 checked 예외를 반드시 선언해야 한다
  2. 자바는 checked 예외를 메서드에 throws로 선언·처리하도록 강제하지만, C#에는 checked 예외 개념이 없어 컴파일러가 강제하지 않는다
  3. 자바와 C# 모두 모든 예외가 unchecked라 선언이 필요 없다
  4. C#은 checked 예외만 있고 unchecked 예외는 없다
정답 확인
② 자바는 checked 예외를 throws로 강제, C#은 강제하지 않음
자바는 checked 예외(예: IOException)를 throws로 선언하거나 잡도록 컴파일러가 강제하지만, C#은 모든 예외가 사실상 unchecked라 선언 강제가 없다. 대신 C#은 문서/주석으로 예외 계약을 알린다. 자바 코드를 이관하면 throws가 사라지므로, 어떤 예외가 던져지는지 팀 문서로 남기지 않으면 호출부가 처리를 누락한다.
Q17객관식 · 인터페이스 기본구현

인터페이스에 구현 코드를 담는 기능의 자바↔C# 대비로 옳은 것은?

  1. 인터페이스에 구현을 넣는 건 자바만 가능하고 C#은 절대 불가능하다
  2. 두 언어 모두 인터페이스에는 어떤 구현도 넣을 수 없다
  3. C#은 인터페이스에 필드와 생성자까지 자유롭게 정의할 수 있다
  4. 자바는 default 메서드(8+), C#은 기본 인터페이스 메서드(C# 8+)로 각각 인터페이스에 기본 구현을 넣을 수 있다
정답 확인
④ 자바 default 메서드(8+) ↔ C# 기본 인터페이스 메서드(8+)
자바 8의 default 메서드처럼 C#도 8부터 기본 인터페이스 메서드로 구현을 넣을 수 있다. 둘 다 다중 구현으로 인한 다이아몬드 문제를 규칙으로 해소한다. 공용 인터페이스에 기본 동작을 추가해 기존 구현체를 깨지 않고 확장하는 실무 기법이다.
Q18객관식 · 상수 const/readonly

자바 final 상수와 C#의 const/readonly 대비로 옳은 것은?

  1. C#의 const는 런타임에 아무 때나 바꿀 수 있다
  2. C#에는 상수 개념이 없어 자바 final만 이식해 쓴다
  3. 자바 컴파일타임 상수는 static final, C#은 const(컴파일타임)와 readonly(런타임·생성자에서만 설정)로 나뉜다
  4. readonly는 컴파일타임 상수라 const와 완전히 동일하다
정답 확인
③ 자바 static final ↔ C# const(컴파일타임)·readonly(런타임)
자바는 static final 하나로 상수를 표현하지만, C#은 const(컴파일 시 인라인되는 진짜 상수)와 readonly(생성자/선언 시 한 번만 설정 가능한 런타임 상수)로 구분한다. 여러 어셈블리로 배포되는 SI 환경에서 const는 값이 참조 측에 박혀 버려, 라이브러리만 재배포해도 값이 안 바뀌는 함정이 있어 공용 설정값엔 readonly/static을 선호한다.
Q19코드읽기 · Dictionary 조회

다음 C# 코드의 출력은? 자바 HashMap.get이 null을 주는 것과 대비하며 읽어라.

var map = new Dictionary<string, int> { ["a"] = 1, ["b"] = 2 };
if (map.TryGetValue("c", out int v))
    Console.WriteLine(v);
else
    Console.WriteLine("none");
모범답안 확인
모범답안 · none
자바 HashMap이라면 get("c")가 null을 반환하지만, C# Dictionary는 인덱서 map["c"]로 없는 키에 접근하면 KeyNotFoundException을 던진다. 그래서 TryGetValue로 존재 여부(bool)와 값(out)을 함께 안전하게 받는다. 코드 매핑·설정 조회에서 이 패턴이 예외 없이 방어적으로 동작하게 한다.
Q20단답 · 진입점 Main

자바 프로그램 진입점은 public static void main(String[] args)이다. C#의 전통적 진입점 메서드 시그니처를 대문자 관례까지 정확히 쓰시오.

모범답안 확인
모범답안
static void Main(string[] args) — 자바는 소문자 main, C#은 대문자 Main(PascalCase)이다.

자바는 진입점이 소문자 main, C#은 PascalCase 관례에 따라 대문자 Main이다. C#은 int 반환·async Task·매개변수 생략 등 여러 형태를 허용한다. 콘솔 배치/유틸을 빠르게 만들 때 진입점 이름 차이(대소문자)를 헷갈리면 컴파일러가 진입점을 못 찾는다.

이 세트 우선복습 3줄

  1. virtual/override 명시 — 자바 메서드는 기본이 재정의 가능하지만, C#은 기반 virtual + 파생 override를 빠뜨리면 재정의가 아니라 숨김(new)이 돼 다형성이 깨진다.
  2. Dictionary 없는 키 조회 — 자바 HashMap.get은 null을 주지만 C# 인덱서 map["c"]는 KeyNotFoundException. 반드시 TryGetValue/ContainsKey로 방어한다.
  3. const vs readonly — const는 컴파일 시 참조 측에 값이 박혀 라이브러리만 재배포해도 안 바뀌는 함정. 공용 설정값은 static readonly를 선호한다.
