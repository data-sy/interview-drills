🗄️ MyBatis · JPA 레퍼런스

SQL 매핑(MyBatis) vs ORM(JPA) · SI/SM에서 둘 다 흔하다 · XML은 별도 색상

MyBatis — SQL 매퍼

  • SQL을 직접 작성 → 쿼리 완전 통제
  • 복잡한 통계/조인/레거시 DB에 강함
  • SQL ↔ 객체 매핑만 담당(ORM 아님)
  • SI 유지보수·금융/공공 현장에 여전히 많음
  • 단점: SQL을 다 짜야 함(반복 많음)

JPA — ORM (보통 Hibernate)

  • 객체 중심 → 기본 CRUD SQL 자동 생성
  • 생산성 높음, 표준(JPA) 기반 이식성
  • 객체 그래프·연관관계 매핑
  • Spring Data JPA로 메서드 이름만으로 쿼리
  • 단점: 러닝커브, N+1·쿼리 예측 필요
한 줄 정리 쿼리를 손으로 쥐고 가야 하는 복잡 SQL·레거시 → MyBatis, 도메인 중심으로 빠르게 CRUD → JPA. 실무에선 둘을 함께 쓰기도 한다(단순 CRUD는 JPA, 복잡 통계는 MyBatis).
MyBatis

01Mapper 인터페이스 + XML

가장 표준적인 방식. 자바 인터페이스(메서드 선언) + XML(SQL). namespace와 인터페이스 경로, id와 메서드명이 연결된다.

① Mapper 인터페이스 (Java)

② Mapper XML (SQL)

면접 포인트
  • namespace는 Mapper 인터페이스의 풀 패키지 경로, id는 메서드명과 일치해야 매핑된다.
  • DB 컬럼(snake_case)과 자바 필드(camelCase)가 다르면 resultMap이나 mapUnderscoreToCamelCase 설정으로 매핑.

02애노테이션 방식 (XML 없이)

간단한 쿼리는 인터페이스에 애노테이션으로 바로. XML 파일이 필요 없다.

면접 포인트
  • 간단 쿼리는 애노테이션, 복잡·동적 SQL은 XML이 유지보수에 유리 — 현장에선 XML이 여전히 주류.

03#{} vs ${} (중요·면접 단골)

MyBatis 파라미터 바인딩의 핵심 차이. 보안(SQL Injection)과 직결된다.

#{ } (권장)${ } (주의)
처리PreparedStatement ? 바인딩문자열 그대로 치환
결과WHERE id = ?WHERE id = 5
SQL Injection안전 ✅취약 ⚠️ (사용자 입력 금지)
용도거의 모든 값 바인딩컬럼명·테이블명·정렬방향 등 구조가 동적일 때만
면접 포인트
  • 왜 #{}가 안전한가: PreparedStatement의 ? 자리표시자로 바인딩되어 값이 SQL 구문으로 해석되지 않기 때문. ${}는 문자열을 그대로 이어붙여 주입 공격에 노출된다.
  • ${}는 컬럼명·정렬 방향 등 파라미터로 바인딩 불가능한 구조에만, 반드시 허용 목록(화이트리스트) 검증 후 사용.

04동적 SQL

조건에 따라 SQL을 조립. <if>, <where>, <foreach>, <choose>가 핵심 태그.

면접 포인트
  • <where>는 첫 조건 앞의 불필요한 AND/OR를 자동 제거 → 직접 WHERE 1=1 트릭을 안 써도 된다.
  • XML에서 부등호 <는 &lt;로 escape하거나 <![CDATA[ ... ]]>로 감싼다.
  • <foreach>는 IN 절, 다건 INSERT에 필수. open/separator/close로 괄호·구분자 처리.

05resultMap (컬럼-필드 매핑)

컬럼명과 필드명이 다르거나 조인 결과를 객체로 조립할 때.

면접 포인트
  • resultType은 컬럼=필드가 일치할 때 간편, resultMap은 이름이 다르거나 조인·중첩 매핑이 필요할 때.
  • 전역 설정 mapUnderscoreToCamelCase=true면 member_name → memberName 자동 매핑.
JPA (Spring Data JPA)

06Entity 매핑

자바 클래스 ↔ DB 테이블. 애노테이션으로 매핑 규칙을 선언한다.

면접 포인트
  • @GeneratedValue 전략: IDENTITY(DB auto_increment, MySQL), SEQUENCE(Oracle 시퀀스), AUTO(방언에 위임).
  • enum은 @Enumerated(EnumType.STRING)로. 기본값 ORDINAL은 순서가 바뀌면 데이터가 깨진다.
  • JPA는 리플렉션으로 객체를 만들기 위해 기본 생성자가 필요.

07연관관계 매핑

객체의 참조와 DB의 FK를 매핑. 방향·주인(owner) 개념이 핵심.

면접 포인트
  • 연관관계의 주인: FK를 가진 쪽(@ManyToOne·@JoinColumn)이 주인. 반대편은 mappedBy로 "나는 주인이 아니다"를 표시(읽기 전용).
  • fetch는 LAZY가 기본 원칙: @ManyToOne·@OneToOne은 기본이 EAGER라 명시적으로 LAZY로 바꾸는 게 관례(N+1·불필요 조회 방지).

08Repository (Spring Data JPA)

JpaRepository를 상속하면 기본 CRUD 메서드가 자동 제공된다. 구현 클래스 불필요.

면접 포인트
  • JpaRepository<T, ID> 상속만으로 CRUD·페이징·정렬 메서드가 자동. 인터페이스만 선언하면 구현은 스프링이 주입.
  • save()는 PK 유무/영속 상태로 INSERT·UPDATE를 판단(merge/persist).

09쿼리 메서드 · @Query

메서드 이름으로 쿼리를 자동 생성하거나, 복잡하면 JPQL/네이티브 SQL을 직접.

면접 포인트
  • 쿼리 메서드: findBy + 필드 + 조건 키워드(And/Or/GreaterThan/Containing/OrderBy)로 SQL 자동 생성. 이름이 곧 스펙.
  • JPQL vs 네이티브: JPQL은 엔티티/필드 기준(DB 독립), 네이티브는 실제 테이블/컬럼 기준(DB 종속). 복잡 통계는 네이티브.
  • @Modifying 없이 UPDATE/DELETE @Query를 쓰면 예외. 벌크 연산은 영속성 컨텍스트를 우회하니 주의.

10페이징 · 정렬

Pageable만 넘기면 페이징 SQL(LIMIT/OFFSET)과 전체 건수 조회를 자동 처리.

면접 포인트
  • Page는 전체 건수 count 쿼리를 추가로 실행, Slice는 count 없이 다음 페이지 유무만(더보기 UI에 적합).
  • 페이지 번호는 0부터 시작.

11N+1 문제 (면접 필수)

JPA 최대 함정. 연관 엔티티를 지연 로딩할 때 목록 조회 1번 + 각 건마다 추가 쿼리 N번이 나가는 현상.

면접 포인트
  • 원인: 지연 로딩된 연관 엔티티를 반복문에서 하나씩 접근할 때, 건별로 추가 쿼리가 발생.
  • 해결: ① fetch join(JPQL JOIN FETCH) ② @EntityGraph ③ default_batch_fetch_size(IN 절로 묶기).
  • 컬렉션(1:N) fetch join은 페이징과 함께 쓰면 문제 → 이땐 @BatchSize/배치 사이즈로 푼다.
  • 기본을 LAZY로 두고, 필요한 조회에서만 fetch join으로 가져오는 게 정석.
