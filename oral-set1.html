OOP·예외·트랜잭션부터 소통·요구분석까지 말로 답하기

질문을 소리 내어 말하듯 답안 칸에 적어본 뒤 모범답변 확인을 눌러 요지와 비교하고, 스스로 O/X로 채점하세요.

Q1구두 · OOP 4대 특성

객체지향의 4대 특성을 각각 설명하고, 왜 그런 특성이 필요한지 말해보세요.

모범답변 요지
  • 캡슐화(encapsulation): 데이터와 기능을 하나로 묶고 내부 구현을 숨겨 외부 영향 최소화
  • 상속(inheritance): 공통 코드를 재사용하고 계층 구조로 확장성 확보
  • 다형성(polymorphism): 같은 메시지에 객체마다 다르게 반응, 유연한 확장
  • 추상화(abstraction): 핵심 개념만 노출해 복잡도를 낮춤
  • 실무 연결: SI는 고객사가 바뀌어도 유지보수·기능 추가가 잦아, 이 4특성이 결국 "변경에 강한 코드"라는 한 줄로 수렴한다는 점을 언급
Q2구두 · 예외처리 전략

실무에서 예외를 처리할 때 어떤 원칙을 지키는지 설명해보세요.

모범답변 요지
  • 잡을 수 있고 복구 가능한 예외만 catch, 아니면 상위로 전파(throws)
  • 예외를 삼키지 않기(빈 catch 금지), 로그에 원인·스택트레이스 남기기
  • 자원 정리는 finally 또는 try-with-resources로 누수 방지
  • 사용자에게는 친화적 메시지, 시스템에는 상세 로그로 분리
  • 실무 연결: 고객사 운영 시스템에서 장애 원인 추적의 시작점이 로그이므로, 예외를 어떻게 남기느냐가 유지보수 속도를 좌우
Q3구두 · 레거시 유지보수

문서도 부족한 오래된 시스템을 처음 넘겨받았을 때 어떤 순서로 접근하겠습니까?

모범답변 요지
  • 먼저 실행 환경·빌드·배포 흐름부터 파악하고 전체 구조를 훑기
  • 함부로 고치지 않고, 로그·DB·화면 흐름을 따라 동작을 관찰
  • 작은 변경부터 테스트하며 영향 범위를 확인, 변경 전후 비교
  • 파악한 내용을 문서로 남겨 다음 사람에게 전달
  • 실무 연결: SI 유지보수는 "돌아가는 시스템을 깨지 않는 것"이 최우선이라, 이해 없이 수정하지 않는 신중함을 강조
Q4구두 · 트랜잭션/ACID

트랜잭션이 무엇이고 ACID가 왜 필요한지 설명해보세요.

모범답변 요지
  • 트랜잭션: 하나의 논리적 작업 단위, 전부 성공(COMMIT) 아니면 전부 취소(ROLLBACK)
  • 원자성(Atomicity)·일관성(Consistency)·격리성(Isolation)·지속성(Durability)
  • 예: 계좌 이체는 출금과 입금이 모두 성공해야 함
  • 실무 연결: 통계·경영정보 시스템에서 금액·재고 같은 데이터 정합성이 깨지면 신뢰를 잃으므로 트랜잭션 경계를 정확히 잡는 것이 중요
Q5구두 · GET vs POST

HTTP의 GET과 POST의 차이를 설명하고 언제 무엇을 쓰는지 말해보세요.

모범답변 요지
  • GET: 데이터 조회, 파라미터가 URL 쿼리스트링에 노출, 캐시·북마크 가능, 멱등
  • POST: 데이터 생성·전송, 본문(body)에 담아 전송, 길이 제한이 사실상 없음
  • 민감정보나 생성/변경 요청은 POST, 단순 조회는 GET
  • 실무 연결: 로그인·개인정보 전송을 GET으로 하면 URL·로그에 노출되어 보안 사고로 이어지므로 메서드 선택이 곧 보안 문제임을 언급
Q6구두 · 협업·소통

기획자나 고객사 담당자 같은 비개발 직군과 일할 때 소통을 어떻게 하겠습니까?

모범답변 요지
  • 기술 용어 대신 상대가 아는 언어로 풀어서 설명
  • 요구를 들은 뒤 내가 이해한 내용을 되짚어 확인(재확인)
  • 화면·표·예시 등 눈에 보이는 것으로 합의
  • 결정 사항은 메일·메신저로 기록해 나중에 근거로 활용
  • 실무 연결: SI는 고객사와의 커뮤니케이션이 개발만큼 중요하며, 구두 합의만 믿으면 나중에 분쟁이 되므로 기록 습관을 강조
Q7구두 · 프로세스 vs 스레드

프로세스와 스레드의 차이를 설명해보세요.

모범답변 요지
  • 프로세스: 실행 중인 프로그램, 독립된 메모리 공간(코드·데이터·힙·스택)
  • 스레드: 프로세스 내 실행 흐름, 힙·코드 등 메모리를 공유하고 스택은 별도
  • 스레드는 컨텍스트 스위칭 비용이 낮지만 공유 자원 동기화 문제 발생
  • 실무 연결: 웹 애플리케이션(WAS)이 다중 요청을 스레드로 처리하므로, 공유 변수 동시 접근 시 데이터가 꼬이는 문제를 이해하는 것이 실무에서 중요
Q8구두 · REST란

REST가 무엇인지, RESTful하다는 것이 어떤 의미인지 설명해보세요.

모범답변 요지
  • 자원(Resource)을 URI로 표현하고, HTTP 메서드(GET/POST/PUT/DELETE)로 행위를 표현
  • 무상태성(stateless): 각 요청은 독립적이고 서버가 상태를 저장하지 않음
  • 자원은 명사로, 행위는 메서드로 구분하는 일관된 규칙을 지킬 때 RESTful
  • 실무 연결: 여러 시스템·클라이언트가 붙는 SI 환경에서 일관된 API 규칙은 협업과 유지보수 비용을 크게 줄인다는 점 언급
Q9구두 · 요구사항 분석

고객이 "이런 기능 만들어 주세요"라고 했을 때, 바로 개발하지 않고 무엇부터 하겠습니까?

모범답변 요지
  • 요구의 목적·배경을 먼저 질문해 진짜 원하는 것을 파악
  • 입력·출력·예외 상황·데이터 범위 등 구체적 조건을 정리
  • 정리한 내용을 문서·화면 예시로 재확인받아 인식 차이 제거
  • 일정·우선순위를 함께 협의
  • 실무 연결: SI에서 요구 오해는 재작업과 납기 지연의 주범이므로, 착수 전 합의와 문서화가 핵심임을 강조
Q10구두 · 왜 SI

여러 분야가 있는데 왜 SI 기업에 지원했나요?

모범답변 요지
  • 다양한 산업·도메인의 시스템을 경험하며 폭넓게 성장하고 싶음
  • 요구 분석부터 개발·운영까지 전 과정을 경험할 수 있는 점에 매력
  • 고객의 실제 문제를 해결하고 결과를 눈으로 확인하는 보람
  • 기본기(DB·백엔드·소통)를 탄탄히 쌓기 좋은 환경이라 판단
  • 실무 연결: 자기 성장 욕구를 회사가 하는 SI 업무의 특성과 연결해, 오래 함께할 사람이라는 인상을 주는 것이 포인트

이 세트 우선복습 3줄

  1. OOP 4대 특성 — 캡슐화·상속·다형성·추상화를 각각 한 줄로 정의하고, 모두 "변경에 강한 코드"로 수렴한다는 실무 연결까지 말할 수 있어야 한다.
  2. 트랜잭션과 ACID — "전부 성공 아니면 전부 취소"라는 원자성 개념과 A·C·I·D 네 글자를, 계좌 이체 예시로 막힘없이 풀어낼 것.
  3. GET vs POST = 보안 문제 — GET은 URL 노출·멱등·조회, POST는 body·생성/변경. 민감정보를 GET으로 보내면 로그에 남는다는 보안 관점이 핵심 포인트.
