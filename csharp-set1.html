Q1객관식 · 명명 규칙

자바와 C#의 관례적 명명 규칙 대비로 옳은 것은?

  1. 자바는 메서드에 camelCase(getName), C#은 public 메서드·프로퍼티에 PascalCase(GetName)를 쓴다
  2. 자바는 PascalCase, C#은 camelCase로 자바와 정반대 방향으로 통일돼 있다
  3. 두 언어 모두 메서드에 snake_case(get_name)를 표준으로 쓴다
  4. 클래스명은 자바만 대문자 시작이고 C#은 소문자로 시작한다
정답 확인
① 자바는 메서드에 camelCase(getName), C#은 public 메서드·프로퍼티에 PascalCase(GetName)를 쓴다
자바 메서드는 camelCase(getName), C#의 public 메서드·프로퍼티는 PascalCase(GetName)가 관례다. 클래스명은 두 언어 모두 PascalCase로 같다. SI 유지보수에서 자바 코드를 C#으로 포팅할 때 식별자 스타일을 일괄 변환해야 코드리뷰 지적을 줄인다.
Q2객관식 · 프로퍼티 vs 필드

자바의 getter/setter와 C# 프로퍼티(property)의 대비로 옳은 것은?

  1. C#에는 캡슐화 개념이 없어 필드를 항상 public으로 열어 둔다
  2. 자바 프로퍼티는 obj.Name처럼 쓰고, C#은 obj.getName()으로 호출한다
  3. 자바는 getName()/setName() 메서드로 캡슐화하고, C#은 Name { get; set; } 프로퍼티 문법으로 같은 목적을 달성한다
  4. 프로퍼티는 자바에만 있고 C#은 getter/setter 메서드만 지원한다
정답 확인
③ 자바는 getName()/setName() 메서드로 캡슐화하고, C#은 Name { get; set; } 프로퍼티 문법으로 같은 목적을 달성한다
자바는 getName()/setName() 메서드로, C#은 프로퍼티 문법(Name { get; set; })으로 필드 접근을 캡슐화한다. 호출 측에서 C#은 obj.Name = x처럼 필드처럼 쓰지만 실제로는 접근자가 실행된다. 고객사 도메인 모델을 옮길 때 자바 getter/setter 뭉치가 C#에선 프로퍼티 한 줄로 줄어든다.
Q3객관식 · var 타입추론

자바의 var와 C#의 var 대비로 옳은 것은?

  1. C# var는 자바스크립트처럼 어떤 타입이든 재대입 가능한 동적 타입이다
  2. 둘 다 지역변수 타입추론이지만, 자바 var는 지역변수 전용, C# var도 지역변수 전용이며 둘 다 런타임 동적 타입이 아니다
  3. 자바에는 var가 아예 없고 C#에만 타입추론이 있다
  4. C# var는 필드 선언에도 자유롭게 쓸 수 있어 자바보다 범위가 넓다
정답 확인
② 둘 다 지역변수 타입추론이지만, 자바 var는 지역변수 전용, C# var도 지역변수 전용이며 둘 다 런타임 동적 타입이 아니다
자바 var(10+)와 C# var는 모두 컴파일 타임에 타입이 확정되는 지역변수 타입추론이며, 선언 후 다른 타입 재대입은 불가하다. 둘 다 필드에는 쓸 수 없다. 실무에서 익명 타입(LINQ 결과 등)을 받을 때 var가 필수라 남용/절제 기준을 팀 컨벤션으로 정해 둔다.
Q4객관식 · 접근제어자 internal

자바의 default(package-private)와 가장 가까운 C# 접근제어자는?

  1. internal — 같은 어셈블리 내에서만 접근 가능
  2. private — 같은 클래스에서만 접근 가능
  3. protected — 파생 클래스에서만 접근 가능
  4. public — 어디서나 접근 가능
정답 확인
① internal — 같은 어셈블리 내에서만 접근 가능
자바에서 제어자를 생략하면 default(같은 패키지 내 접근)이고, C#에는 default가 없는 대신 internal(같은 어셈블리 내 접근)이 유사한 역할을 한다. 단위가 패키지↔어셈블리로 다른 점이 함정이다. SI에서 공용 라이브러리를 별 DLL로 뽑을 때 internal 경계 설계가 API 노출 범위를 좌우한다.
Q5객관식 · string 불변성

자바 String과 C# string의 불변성(immutability) 대비로 옳은 것은?

  1. 자바 String은 불변이지만 C# string은 가변이라 내부 문자 배열을 바꿀 수 있다
  2. 둘 다 가변이라 반복 연결에도 성능 문제가 없다
  3. C# string은 값 형식이라 불변, 자바 String은 참조 형식이라 가변이다
  4. 자바 String과 C# string 모두 불변이라 문자열 조작 시 새 인스턴스가 생성된다
정답 확인
④ 자바 String과 C# string 모두 불변이라 문자열 조작 시 새 인스턴스가 생성된다
자바 String과 C# string은 둘 다 참조 형식이면서 불변이라, + 연결이 반복되면 매번 새 객체가 생긴다. 그래서 두 언어 모두 반복 연결엔 StringBuilder를 쓴다. 통계/배치 시스템에서 대량 리포트 문자열을 만들 때 불변성 무시가 GC 부담과 성능 저하로 직결된다.
Q6객관식 · 네임스페이스 vs 패키지

자바 package와 C# namespace 대비로 옳은 것은?

  1. C# namespace는 파일당 하나만 선언 가능하고 자바 package는 여러 개 선언 가능하다
  2. 자바 package는 폴더 구조와 강하게 결합되지만, C# namespace는 폴더 구조와 반드시 일치할 필요가 없다
  3. 자바는 import, C#은 include로 다른 패키지를 가져온다
  4. namespace는 자바에만 있고 C#은 package 키워드를 쓴다
정답 확인
② 자바 package는 폴더 구조와 강하게 결합되지만, C# namespace는 폴더 구조와 반드시 일치할 필요가 없다
자바 package는 소스 폴더 경로와 일치해야 하지만, C# namespace는 물리적 폴더와 무관하게 논리적으로 선언할 수 있다. 가져오기는 자바 import ↔ C# using이다(C의 include 아님). 대규모 SI 솔루션에서 namespace 규칙을 폴더와 맞춰 두면 신규 투입 인력이 코드를 빨리 찾는다.
Q7객관식 · List<T> vs ArrayList

자바 컬렉션과 C# 컬렉션의 대응으로 옳은 것은?

  1. 자바 ArrayList ↔ C# ArrayList<T>로 이름과 타입 파라미터가 완전히 동일하다
  2. C#에는 제네릭 리스트가 없어 항상 object 배열로 다룬다
  3. 자바 ArrayList<E> ↔ C# List<T>, 둘 다 내부적으로 배열 기반 동적 리스트다
  4. 자바 List는 인터페이스지만 C# List<T>는 존재하지 않는 타입이다
정답 확인
③ 자바 ArrayList<E> ↔ C# List<T>, 둘 다 내부적으로 배열 기반 동적 리스트다
자바 ArrayList<E>에 대응하는 C#의 표준 동적 리스트는 List<T>이며 둘 다 배열 기반이다. C#의 비제네릭 ArrayList는 구형(박싱 유발)이라 실무에서는 List<T>를 쓴다. 고객사 데이터 목록을 다룰 때 제네릭 List<T>로 타입 안전성을 지켜 런타임 캐스팅 오류를 막는다.
Q8객관식 · 상속 문법

클래스 상속·인터페이스 구현 문법의 자바↔C# 대비로 옳은 것은?

  1. 자바는 extends/implements 키워드를, C#은 둘 다 콜론(:)으로 표기한다
  2. 자바와 C# 모두 extends 키워드로 통일돼 있다
  3. C#은 다중 클래스 상속을 지원해 자바와 근본적으로 다르다
  4. C#은 인터페이스 구현에 implements를 쓰고 상속엔 콜론을 쓴다
정답 확인
① 자바는 extends/implements 키워드를, C#은 둘 다 콜론(:)으로 표기한다
자바는 class B extends A implements I처럼 키워드로 구분하지만, C#은 class B : A, I로 클래스·인터페이스 모두 콜론으로 나열한다. 단일 클래스 상속 + 다중 인터페이스 구현 원칙은 두 언어가 같다. 레거시 클래스 계층을 이관할 때 콜론 뒤 첫 항목이 기반 클래스라는 규칙을 지켜야 컴파일된다.
Q9코드읽기 · 자동 프로퍼티

다음 C# 코드에서 출력값은? 자바의 getter/setter와 대비하며 읽어라.

class Person {
    public string Name { get; set; }
    public int Age { get; private set; }
    public Person(int age) { Age = age; }
}
var p = new Person(20) { Name = "Kim" };
Console.WriteLine($"{p.Name}/{p.Age}");
모범답안 확인
모범답안 · Kim/20
자바라면 name/age 필드 + getName()/getAge()로 썼을 것을, C#은 자동 프로퍼티로 축약한다. Age는 set이 private이라 외부에서 못 바꾸고 생성자에서만 설정된다. 객체 초기화자 { Name = "Kim" }는 public set이 있어 허용된다. 도메인 불변식(나이 외부 변경 금지)을 프로퍼티 접근자 수준에서 강제하는 실무 패턴이다.
Q10단답 · foreach 문법

자바의 향상된 for문 for (String s : list)에 대응하는 C#의 컬렉션 순회 반복문을 키워드까지 정확히 쓰시오.

모범답안 확인
모범답안
foreach (string s in list) — 자바는 콜론(:), C#은 in 키워드를 사용한다.

자바 for-each는 for (T x : coll), C#은 foreach (T x in coll)로 콜론 대신 in을 쓴다. 둘 다 내부적으로 Iterator/IEnumerator를 돌린다. 대량 조회 결과를 순회 처리하는 배치 코드에서 가장 자주 쓰는 구문이라 구분을 몸에 익혀야 한다.

이 세트 우선복습 3줄

  1. internal ↔ default — 자바 default(패키지 단위)에 대응하는 C# 키워드는 internal(어셈블리 단위). 단위가 패키지↔어셈블리로 다른 게 함정.
  2. 프로퍼티 vs getter/setter — 자바 getName()/setName() 뭉치가 C#에선 Name { get; set; } 한 줄. 호출은 필드처럼 보여도 접근자가 실행된다.
  3. string 불변성은 양쪽 공통 — 자바 String·C# string 모두 참조형이면서 불변. 반복 연결은 두 언어 다 StringBuilder로.
